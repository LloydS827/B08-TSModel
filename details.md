# B08 设备时序基础模型进展说明

更新日期：2026-06-02

## 一句话说明

这个项目正在做一件事：为设备预测性维护项目建立一个“底层时序模型核心”，让研发人员能够判断设备传感器数据是否可以被模型理解、预测、补全、表征，并进一步判断是否可以直接使用开源时序大模型，还是需要微调或自研。

它不是一个完整的告警系统，也不是一个已经训练好的设备大模型。它目前是一个可运行的研发沙盒，用来把数据、模型输入输出、评测方法和后续路线先跑通。

## FU13 是什么

FU13 是本课题当前选定的第一台示例设备编号，指一台真空速凝炉。

在项目里，它有三个作用：

- 作为真实或模拟数据中的设备 ID。
- 作为模型输入输出定义的第一个具体场景。
- 作为开源模型和 baseline 评测时的示例设备对象。

它不是算法名，也不是模型名。后续接入更多设备时，FU13 只是 `device_id` 中的一个设备编号。

## 为什么要做

设备预测性维护最终要回答的问题是：设备是否正在变差、未来是否可能出问题、什么时候需要干预。

但在真正做风险判断之前，必须先解决更底层的问题：

- 设备传感器数据能不能整理成模型能吃的格式。
- 模型能不能理解不同工艺阶段下的传感器变化。
- 模型能不能预测未来一段时间的正常轨迹。
- 模型能不能发现异常、缺失、漂移和退化迹象。
- 现有开源时序模型是否已经足够，还是需要做领域适配。

本项目当前就是在解决这些基础问题。

## 当前阶段判断

当前阶段：**FU13 真实数据 pipeline 已跑通，当前正在标准化 TTM 真实数据评测和开发者 workflow**。

这里的“跑通”指的是：已经具备一套可运行的研发工作台，可以生成模拟数据、整理模型输入、跑基础预测评测、装配真实现场导出数据，并在真实窗口上完成 baseline 与 TTM 的同口径验证。当前工作重点已经从“数据链路能否运行”转向“把 TTM 真实数据能力复核、标准评测口径和开发者复现流程固化下来”。

它不代表预测性维护项目已经交付，也不代表设备大模型已经训练完成。当前已经确认“现场导出 -> 标准观测表 -> 数据诊断 -> 模型窗口 -> baseline/TTM 对比”这条工程流程可以跑通；后续将在评测口径清楚后，进入模型开发、模型选择、轻量微调或训练路线判断。

## 已经具备的能力

### 1. 模拟设备数据

项目已经可以生成 FU13 真空速凝炉风格的模拟时序数据。

这套模拟数据用于在没有完整真实数据之前，先搭建模型输入、窗口构建、评测指标和实验流程。

已有能力：

- 模拟 45 天左右的设备运行数据。
- 覆盖多个工艺阶段。
- 覆盖多类传感器和物理域。
- 可以生成用于模型实验的 parquet 数据文件。

### 2. 模型输入格式定义

项目已经定义了模型需要看到的基础数据格式。

简单说，模型不是直接读取“业务系统里的所有信息”，而是先把数据转成统一的观测表，再切成时序窗口。

统一观测表包含：

- 时间。
- 设备。
- 批次。
- 工艺阶段。
- 传感器。
- 数值。
- 单位。
- 物理域。
- 数据质量标记。
- 弱退化标签。
- 弱故障代理标签。

这一步很关键，因为它决定后续无论使用开源模型、微调模型还是自研模型，都有统一的数据入口。

### 3. 真实数据验证入口

项目已经有了真实数据验证命令，并新增了 FU13 专用真实数据闭环命令。

它的作用不是训练模型，而是检查真实导出的设备数据能不能映射成模型需要的标准格式，并进一步形成可用于诊断和 forecasting 的标准观测表。

当前可以检查的问题包括：

- 时间字段是否能正确解析。
- 传感器数值是否真的是数值。
- 是否存在重复点。
- 是否存在未知传感器。
- 是否存在未映射的工艺阶段。
- 宽表数据中是否缺少配置过的传感器列。
- 缺失值是否会影响模型输入。
- FU13 多 CSV 导出能否按 8 个传感器装配。
- 连续炉工艺阶段能否重构为 cycle。
- 各场景和各传感器是否有足够窗口进入模型验证。

验证结果会生成一份报告。数据合格时命令返回成功；数据有问题但能完成检查时，会写出报告并返回失败状态，方便研发人员定位问题。

### 4. 基础预测评测

项目已经有基础预测 baseline。

baseline 的作用是提供一个最低对照：后续任何开源模型或微调模型，都应该和这个基线比较，而不是只看模型名字是否先进。

已有能力：

- 构建模型窗口。
- 跑阶段感知预测 baseline。
- 输出 MAE、区间覆盖率等评测指标。
- 生成模型路线评估摘要。

### 5. 第一轮 forecasting 实验脚手架

项目已经可以运行第一轮 forecasting 实验脚手架。

这一步已经完成第一条真实基础模型推理链路：在本机安装 TTM optional 依赖，下载 IBM Granite TTM R2 权重到本机 cache，并分别在 FU13 模拟窗口和 FU13 真实数据窗口上完成 zero-shot forecasting。

当前已经补充基础模型实验的 Markdown 报告渲染能力，可以把数据摘要、选定模型、运行状态、baseline 对比、本机依赖和权重状态、输入输出覆盖情况以及路线建议整理成统一报告。默认 baseline 模式不会下载外部模型权重；只有显式选择 TTM 并允许下载时，才会把权重保存到本机 cache。

forecasting 实验命令现在可以显式选择 `baseline` 或 `ttm`。默认仍然只跑 baseline，不需要 TTM 依赖，也不会下载模型权重；选择 TTM 且数据窗口构建成功时，会写出同一格式的基础模型实验报告，并根据 TTM 是否真实运行成功返回成功或失败状态。已验证的 TTM R2 运行配置是 `context_length=90`、`prediction_length=16`。

### 6. FU13 真实数据闭环

项目已经可以把 `data/real/` 中的 FU13 现场导出数据装配为 canonical observation 数据，并完成数据诊断、baseline forecasting 和 TTM forecasting。

本轮真实数据验证结果：

- 装配得到 4,126,789 行标准观测。
- 覆盖 8 个传感器和 8 个工艺阶段。
- 重构 428 个 cycle，其中 247 个完整 cycle。
- 识别出 `good`、`unassigned_cycle`、`invalid` 三类质量标记。
- 在真实窗口上完成 baseline 与 TTM 对比；TTM 状态为 `available_and_ran`。
- 本阶段已经输出按场景拆分的指标，但还没有实现按单一场景过滤建窗。
- 当前 cycle 重构会把等待态归入相邻重构 cycle；是否从训练和评测窗口中剔除，放到下一阶段算法治理。

当前关注的第一批候选包括：

- FlowState。
- IBM TinyTimeMixer / TTM。
- TimesFM。
- Chronos。
- Moirai / Uni2TS。

### 7. 知识成果规划

项目已经把专利、论文、benchmark、模型资产等知识成果纳入规划。

这很重要，因为本课题不仅是做一个工程模块，也需要沉淀可以复用、可以申报、可以发表、可以对外说明的知识资产。

### 8. Python 环境可复现

项目已经改为建议使用 `uv` 管理 Python 环境。

已有能力：

- 使用 `uv sync --extra dev` 同步开发环境。
- 使用 `uv run ...` 运行测试和 CLI 命令。
- 提交 `uv.lock` 锁定 Python 依赖解析结果，减少不同机器上的环境差异。

## 目前还没有具备的能力

### 1. 还没有完成算法效果提升和多模型扩展

当前已经在真实 FU13 数据上完成 baseline 与 TTM 的同口径验证，但还没有完成 TimesFM、Chronos、Moirai、FlowState 等更多模型的真实 zero-shot/frozen inference 对比，也还没有做针对 FU13 工艺阶段和传感器特征的模型适配。

下一阶段需要把重点从“流程能否跑通”转向“哪些模型和特征组合真正有效”。

### 2. 还没有真实故障标签和维护闭环

当前真实数据可以产生候选异常信号和质量诊断，但还没有接入维修记录、故障事件、停机原因或 RUL 标签。

因此现阶段不能声称真实故障预测、风险概率或维护建议，只能评价数据质量、窗口预测和异常候选信号。

### 3. 还没有形成生产级预测性维护功能

项目目前不会直接生成工单、维护建议、日报或正式告警。

这些属于上层业务系统能力，需要在底层模型验证有效之后再接入。

### 4. 还没有训练自研设备时序基础模型

自研基础模型是可能的后续路线，但不是当前阶段的默认选择。

当前更合理的顺序是：先用真实数据继续评估开源模型和工程特征是否有效，再决定直接引用、轻量微调或领域自研。

## 下一阶段计划

### 第一步：业务场景评测口径固化

目标：在已跑通的 FU13 真实数据闭环上，建立从基础模型输出到业务候选信号的评测桥梁。

这一步不是直接做生产告警，也不是马上接入更多模型。它要先回答：某个业务场景中，模型应该看哪些传感器、哪些工艺阶段、哪些窗口；模型输出的误差、残差、趋势或尖峰，哪些可以作为候选异常信号；哪些结论还必须等待维修记录、专家复核或真实故障标签验证。

要做的事：

- 先选择漏液电流监测做最小闭环，炉内气氛检测作为第二候选场景。
- 为该场景明确相关传感器、相关工艺阶段和有效窗口规则。
- 分析 `unassigned_cycle`、`invalid`、等待态和异常采样间隔对窗口质量的影响。
- 比较 `good only`、去除 `invalid`、去除 `unassigned_cycle` 等质量口径下的指标变化。
- 将 TTM 与现有 baseline、一个 rolling 或 lag baseline 做同口径对比。
- 把 forecasting 输出进一步转化为残差、趋势、尖峰或分位数候选信号。
- 明确哪些信号只是候选异常，哪些需要专家复核或维修记录对齐。

完成标准：

- 选定业务场景的窗口选择规则稳定可复用。
- baseline、TTM 和工程特征在同一业务场景口径下可比较。
- 能说明模型输出如何转成候选异常信号。
- 能说明当前还不能推出故障概率、RUL 或维护建议。
- 能给出下一轮模型适配、数据补充或业务验证的明确优先级。

### 第二步：更多 forecasting 模型实测

目标：判断开源时序模型是否可以直接作为项目模块引用，并避免只绑定一个模型生态。

优先测试：

- TTM 或同类轻量模型。
- FlowState。
- TimesFM。
- Chronos。
- Moirai。

完成标准：

- 在同一批真实数据窗口上运行。
- 和现有 baseline 比较 MAE、区间覆盖率等指标。
- 记录依赖成本、部署难度和输入输出适配问题。

### 第三步：表征、补全和异常能力评估

目标：判断模型是否能帮助发现设备状态变化，而不仅是预测未来数值。

优先测试：

- MOMENT。
- TSPulse。
- UniTS。
- 工程基线方法，例如 EWMA、CUSUM、变点检测。

完成标准：

- 能做缺失补全。
- 能生成窗口 embedding。
- 能区分工艺阶段和设备状态。
- 能对退化或异常片段产生更高响应。

### 第四步：路线决策

目标：决定后续采用哪条技术路线。

可能路线：

| 路线 | 什么时候选择 |
| --- | --- |
| 直接引用开源模型 | 开源模型效果好，输入输出适配成本低 |
| 轻量微调 | 开源模型有用，但需要适配工艺阶段和设备传感器 |
| 领域自研 | 开源模型无法满足核心任务，且真实数据规模足够 |

## 对非技术人员的理解方式

可以把当前项目理解为在建设“设备数据进入智能模型之前的发动机和仪表盘”。

现在已经有了：

- 可以试车的模拟道路。
- 统一的数据接口。
- 初步仪表盘。
- 基础参照模型。
- 开源模型测试入口。
- 第一辆真实车辆的上路测试记录。

接下来需要：

- 调校真实仪表的读数质量。
- 比较不同发动机方案在真实道路上的表现。
- 决定是直接买成熟发动机，还是改装，还是自己研发。

## 当前风险

### 真实数据风险

真实数据可能存在字段不一致、采样不稳定、阶段缺失、传感器命名混乱、维护记录无法对齐等问题。

这个风险已经开始被真实数据 validation 暴露出来：当前存在未归入 cycle 和越限/无效读数。后续要判断这些信号分别来自工艺等待、采样问题、维护行为还是设备真实异常。

### 开源模型适配风险

开源模型大多擅长预测，但不一定理解设备工艺阶段、物理域、维修记录和退化前兆。

因此不能只看模型名气，必须用本项目的任务和数据评测。

### 自研模型风险

自研基础模型需要大量数据、算力和长期维护。当前不建议直接进入自研，除非开源路线被验证不足。
