# B08 设备时序基础模型进展说明

更新日期：2026-06-04

## 一句话说明

这个项目正在做一件事：为设备预测性维护项目建立一个“底层时序模型核心”，让研发人员能够判断设备传感器数据是否可以被模型理解、预测、补全、表征，并进一步判断是否可以直接使用开源时序大模型，还是需要微调或自研。

它不是一个完整的告警系统，也不是一个已经训练好的设备大模型。它目前是一个可运行的研发沙盒，用来把数据、模型输入输出、评测方法和后续路线先跑通。

## FU13 是什么

FU13 是本课题当前选定的第一台示例设备编号，指一台真空速凝炉。

在项目里，它有三个作用：

- 作为真实或模拟数据中的设备 ID。
- 作为模型输入输出定义的第一个具体场景。
- 作为开源模型和 baseline 评测时的示例设备对象。

它不是算法名，也不是模型名。后续接入更多设备时，FU13 只是 `device_id` 中的一个设备编号。

## 为什么要做

设备预测性维护最终要回答的问题是：设备是否正在变差、未来是否可能出问题、什么时候需要干预。

但在真正做风险判断之前，必须先解决更底层的问题：

- 设备传感器数据能不能整理成模型能吃的格式。
- 模型能不能理解不同工艺阶段下的传感器变化。
- 模型能不能预测未来一段时间的正常轨迹。
- 模型能不能发现异常、缺失、漂移和退化迹象。
- 现有开源时序模型是否已经足够，还是需要做领域适配。

本项目当前就是在解决这些基础问题。

## 当前阶段判断

当前阶段：**真实数据 pipeline、TTM 真实评测和第一个业务场景评测样例已经完成，下一步主线回到设备时序基础模型研发路线判断**。

这里的“完成”指的是：已经具备一套可运行的研发工作台，可以生成模拟数据、整理模型输入、装配 FU13 真实现场导出数据，并在真实窗口上完成 baseline 与 TTM 的同口径 forecasting 验证；`leak_current_monitoring` 也已经形成第一个 scenario-filtered evaluation（按场景过滤评测）样例，把模型输出转化为 residual candidate signal（残差候选信号）。

它不代表预测性维护项目已经交付，也不代表设备大模型已经训练完成。当前已经确认“现场导出 -> canonical observations -> 数据诊断 -> 模型窗口 -> baseline/TTM 对比 -> 候选信号样例”这条基础设施链路可以跑通。下一步主线从业务场景闭环回到设备时序基础模型研发：先补学术与行业路线判断，再系统适配开源模型，最后决定是否进入自研训练方案设计。

## 已经具备的能力

### 1. 模拟设备数据

项目已经可以生成 FU13 真空速凝炉风格的模拟时序数据。

这套模拟数据用于在没有完整真实数据之前，先搭建模型输入、窗口构建、评测指标和实验流程。

已有能力：

- 模拟 45 天左右的设备运行数据。
- 覆盖多个工艺阶段。
- 覆盖多类传感器和物理域。
- 可以生成用于模型实验的 parquet 数据文件。

### 2. 模型输入格式定义

项目已经定义了模型需要看到的基础数据格式。

简单说，模型不是直接读取“业务系统里的所有信息”，而是先把数据转成统一的观测表，再切成时序窗口。

统一观测表包含：

- 时间。
- 设备。
- 批次。
- 工艺阶段。
- 传感器。
- 数值。
- 单位。
- 物理域。
- 数据质量标记。
- 弱退化标签。
- 弱故障代理标签。

这一步很关键，因为它决定后续无论使用开源模型、微调模型还是自研模型，都有统一的数据入口。

### 3. 真实数据验证入口

项目已经有了真实数据验证命令，并新增了 FU13 专用真实数据闭环命令。

它的作用不是训练模型，而是检查真实导出的设备数据能不能映射成模型需要的标准格式，并进一步形成可用于诊断和 forecasting 的标准观测表。

当前可以检查的问题包括：

- 时间字段是否能正确解析。
- 传感器数值是否真的是数值。
- 是否存在重复点。
- 是否存在未知传感器。
- 是否存在未映射的工艺阶段。
- 宽表数据中是否缺少配置过的传感器列。
- 缺失值是否会影响模型输入。
- FU13 多 CSV 导出能否按 8 个传感器装配。
- 连续炉工艺阶段能否重构为 cycle。
- 各场景和各传感器是否有足够窗口进入模型验证。

验证结果会生成一份报告。数据合格时命令返回成功；数据有问题但能完成检查时，会写出报告并返回失败状态，方便研发人员定位问题。

### 4. 基础预测评测

项目已经有基础预测 baseline。

baseline 的作用是提供一个最低对照：后续任何开源模型或微调模型，都应该和这个基线比较，而不是只看模型名字是否先进。

已有能力：

- 构建模型窗口。
- 跑阶段感知预测 baseline。
- 输出 MAE、区间覆盖率等评测指标。
- 生成模型路线评估摘要。

### 5. 第一轮 forecasting 实验脚手架

项目已经可以运行第一轮 forecasting 实验脚手架。

这一步已经完成第一条真实基础模型推理链路：在本机安装 TTM optional 依赖，下载 IBM Granite TTM R2 权重到本机 cache，并分别在 FU13 模拟窗口和 FU13 真实数据窗口上完成 zero-shot forecasting。

当前已经补充基础模型实验的 Markdown 报告渲染能力，可以把数据摘要、选定模型、运行状态、baseline 对比、本机依赖和权重状态、输入输出覆盖情况以及路线建议整理成统一报告。默认 baseline 模式不会下载外部模型权重；只有显式选择 TTM 并允许下载时，才会把权重保存到本机 cache。

forecasting 实验命令现在可以显式选择 `baseline` 或 `ttm`。默认仍然只跑 baseline，不需要 TTM 依赖，也不会下载模型权重；选择 TTM 且数据窗口构建成功时，会写出同一格式的基础模型实验报告，并根据 TTM 是否真实运行成功返回成功或失败状态。已验证的 TTM R2 运行配置是 `context_length=90`、`prediction_length=16`。

### 6. FU13 canonical data pipeline 与真实数据闭环

项目已经可以把 `data/real/` 中的 FU13 现场导出数据装配为 canonical observations，并完成数据诊断、Baseline / TTM real-data forecasting。

本轮真实数据验证结果：

- 装配得到 4,126,789 行标准观测。
- 覆盖 8 个传感器和 8 个工艺阶段。
- 重构 428 个 cycle，其中 247 个完整 cycle。
- 识别出 `good`、`unassigned_cycle`、`invalid` 三类质量标记。
- 在真实窗口上完成 baseline 与 TTM 对比；TTM 状态为 `available_and_ran`。
- 真实数据闭环阶段最初只输出按场景拆分的指标；现在 `leak_current_monitoring` 已补上第一版场景过滤评测，但跨场景复用和更系统的模型路线评测仍待后续完成。
- 当前 cycle 重构会把等待态归入相邻重构 cycle；是否从训练和评测窗口中剔除，放到下一阶段算法治理。

当前关注的第一批候选包括：

- FlowState。
- IBM TinyTimeMixer / TTM。
- TimesFM。
- Chronos。
- Moirai / Uni2TS。

### 7. `leak_current_monitoring` 业务场景评测样例

项目已经完成 `leak_current_monitoring` 的第一版 scenario-filtered evaluation。

它的定位是验证样例：检查基础模型输出能否在特定设备场景中转化为 residual、趋势或尖峰等 candidate signal，而不是直接给出生产告警、故障概率或维修建议。

已有能力：

- 按漏液电流监测场景过滤相关窗口。
- 在同一场景口径下比较 baseline 与 TTM forecasting 输出。
- 将 forecasting residual 汇总为候选异常信号样例。
- 明确当前信号仍需要专家复核、维修记录或现场维护闭环承接。

### 8. 知识成果规划

项目已经把专利、论文、benchmark、模型资产等知识成果纳入规划。

这很重要，因为本课题不仅是做一个工程模块，也需要沉淀可以复用、可以申报、可以发表、可以对外说明的知识资产。

### 9. Python / uv workflow 可复现

项目已经改为建议使用 `uv` 管理 Python 环境。

已有能力：

- 使用 `uv sync --extra dev` 同步开发环境。
- 使用 `uv run ...` 运行测试和 CLI 命令。
- 提交 `uv.lock` 锁定 Python 依赖解析结果，减少不同机器上的环境差异。

## 目前还没有具备的能力

### 1. 还没有完成学术 / 行业路线调研

当前还没有系统完成设备时序基础模型的学术资料、行业路线和任务谱系调研。

下一阶段需要先回答设备时序基础模型应该学习什么、评测什么，以及它与通用时间序列模型、预测性维护工程系统之间的边界。

### 2. 还没有完成开源基础时序模型系统对比

当前已经在真实 FU13 数据上完成 baseline 与 TTM 的同口径验证，但还没有完成 TimesFM、Chronos、Moirai、FlowState、MOMENT、UniTS 等更多开源模型的真实 zero-shot/frozen inference、embedding、imputation 或 anomaly response 对比，也还没有形成统一 adapter contract 和多模型评测矩阵。

下一阶段需要把重点从“单个模型能否跑通”转向“哪些开源模型和特征组合在同一批设备窗口、同一套指标、同一套报告口径下真正有效”。

### 3. 还没有设计自研设备时序基础模型

自研基础模型是 A/C 之后的条件性路线，但不是当前已经决定的路线。

项目目前还没有设计自研设备时序基础模型的输入格式、预训练目标、训练/验证切分、数据规模需求、算力预算、最小原型或 Go / No-Go 条件。只有当学术/行业路线调研和开源模型系统对比证明现有模型无法覆盖核心缺口时，才应进入自研训练方案设计。

### 4. 还没有形成生产级预测性维护系统

项目目前不是生产级预测性维护系统，不会直接生成工单、维护建议、日报或正式告警。

当前真实数据可以产生数据质量诊断、窗口预测指标和候选异常信号，但还没有接入维修记录、故障事件、停机原因、RUL 标签或现场维护闭环。因此现阶段不能声称真实故障预测、不能给出故障概率或维护建议。

维修记录、生产告警和现场维护闭环属于工程应用侧限制与后续承接事项，可以由工程团队在模型候选信号稳定后继续接入；它们不应被写成本项目下一阶段唯一 blocker。

## 下一阶段计划

### A. 学术 / 行业 / 模型路线调研

目标：先补路线调研，回答设备时序基础模型应训练什么、评测什么，以及它与通用时间序列模型和预测性维护工程系统的差异。

要做的事：

- 梳理 forecasting、imputation、representation、anomaly、classification、RUL 等任务谱系。
- 总结工业设备数据特点，包括多传感器、工艺阶段、非均匀采样、维护事件缺失、工况切换和质量标记。
- 阅读基础时序模型、工业时间序列、设备健康管理和预测性维护相关学术 / 行业资料。
- 建立开源模型能力矩阵，覆盖 TTM、MOMENT、Chronos、TimesFM、Moirai、UniTS 等候选模型。
- 判断现有 FU13 数据能支撑哪些模型研发问题，哪些数据需求属于工程团队后续业务应用承接。

完成标准：

- 能说明为什么需要或不需要自研设备时序基础模型。
- 能给出开源模型优先评测列表。
- 能给出本项目专属的任务、指标和数据需求清单。

### C. 开源基础时序模型系统适配与对比

目标：在 A 的路线判断之后，系统验证已有开源基础时序模型能否作为本项目核心基座。

要做的事：

- 建立统一 adapter contract。
- 统一窗口、指标和报告口径。
- 在同一批设备窗口上比较 forecasting、embedding、imputation、anomaly response 等任务表现。
- 记录依赖成本、部署难度、推理速度、缓存策略和输入输出限制。
- 输出 Go / No-Go 结论：直接引用、轻量微调，还是进入自研训练方案设计。

完成标准：

- 不只比较模型名字，而是在同一数据、同一指标、同一报告口径下比较。
- 能解释模型失败是依赖问题、窗口形状问题、任务不匹配，还是模型能力不足。
- 能给出是否进入自研训练方案设计的证据。

### B. 自研设备时序基础模型训练方案设计

目标：B 是 A/C 之后的条件性路线；只有当路线调研和开源模型系统对比证明现有模型不能覆盖核心缺口时，才进入自研设备时序基础模型训练方案设计。

本阶段不应一开始就大规模训练，而应先形成可审查的训练方案。

要做的事：

- 设计模型输入格式。
- 明确预训练目标。
- 规划数据集组织方式。
- 制定训练 / 验证切分。
- 建立与 baseline 和开源模型的对照口径。
- 形成最小训练原型计划。
- 估算算力、数据规模和维护成本。

完成标准：

- 能说明自研模型相对开源模型要解决的具体缺口。
- 能说明最小训练原型的成功标准。
- 能说明继续投入训练的 Go / No-Go 条件。

## 对非技术人员的理解方式

可以把当前项目理解为在建设“设备数据进入智能模型之前的发动机和仪表盘”。

现在已经有了：

- 可以试车的模拟道路。
- 统一的数据接口。
- 初步仪表盘。
- 基础参照模型。
- 开源模型测试入口。
- 第一辆真实车辆的上路测试记录。

接下来需要：

- 调校真实仪表的读数质量。
- 比较不同发动机方案在真实道路上的表现。
- 决定是直接买成熟发动机，还是改装，还是自己研发。

## 当前风险

### 真实数据风险

真实数据可能存在字段不一致、采样不稳定、阶段缺失、传感器命名混乱、维护记录无法对齐等问题。

这个风险已经开始被真实数据 validation 暴露出来：当前存在未归入 cycle 和越限/无效读数。后续要判断这些信号分别来自工艺等待、采样问题、维护行为还是设备真实异常。

### 开源模型适配风险

开源模型大多擅长预测，但不一定理解设备工艺阶段、物理域、维修记录和退化前兆。

因此不能只看模型名气，必须用本项目的任务和数据评测。

### 自研模型风险

自研基础模型需要大量数据、算力和长期维护。当前不建议直接进入自研，除非开源路线被验证不足。
