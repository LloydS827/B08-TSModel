# 模型输入输出与架构设计

版本：v0.1  
日期：2026-05-30

## 1. 设计原则

B08 的模型不应直接等同于最终业务报告。模型负责输出预测性维护所需的中间证据和风险变量；应用系统负责将其转成告警、日报、工单和维护建议。

核心原则：

- 以未来损坏风险为目标。
- 以时序预测、重构和表征作为模型手段。
- 以阶段、设备和物理域作为上下文。
- 以证据链和可校准风险作为交付出口。

## 2. 输入定义

### 2.1 原始输入

| 输入 | 说明 |
| --- | --- |
| 时间戳 | 原始采样时间，约 5 秒/点 |
| FValue | 传感器数值 |
| 设备标识 | measurement 或设备 ID |
| 传感器标签 | Attribute，如 PumpShake1、SysSelfPressure |
| 工艺阶段 | PLC 阶段信号，映射为 8 个阶段 |
| 设备配置 | 传感器映射、物理域、阈值、知识库 ID |

### 2.2 模型样本输入

```text
X: B × L × C，历史窗口内的多变量传感器值
M: B × L × C，缺失值、停机段和数据质量 mask
S: B × L，工艺阶段 token
D: B，设备 token
A: C，传感器/物理域 token
T: B × L × K，时间间隔、阶段位置、炉次位置等时间特征
P: C × R，物理阈值、量纲、上下限等配置特征
```

其中：

- B 是 batch。
- L 是历史窗口长度。
- C 是传感器数量。
- K 是时间特征数量。
- R 是传感器配置特征数量。

### 2.3 样本组织方式

建议同时支持三种样本：

1. 单传感器阶段样本：适合冷启动和解释。
2. 子系统多变量样本：适合机械、液压、热、气氛、电气、流体等子系统。
3. 设备级多变量样本：适合成熟阶段做综合风险推演。

MVP 阶段优先实现 1 和 2。

## 3. 输出定义

### 3.1 模型原始输出

| 输出 | 作用 |
| --- | --- |
| `y_hat` | 未来窗口预测值 |
| `q_low / q_high` | 预测分位数或置信区间 |
| `recon_x` | 当前窗口重构值 |
| `embedding` | 窗口状态表征 |
| `residual_score` | 预测残差异常度 |
| `reconstruction_score` | 重构异常度 |
| `drift_score` | 相对历史基线的漂移 |
| `change_point_score` | 状态突变可能性 |

### 3.2 风险推演输出

| 输出 | 说明 |
| --- | --- |
| `damage_risk_score` | 未来窗口内预损坏风险分数，0-1 |
| `risk_level` | NORMAL、WATCH、WARNING、ALERT |
| `horizon` | 风险预测窗口，如 next_stage、next_batch、24h、7d |
| `eta` | 若趋势持续，预计到达危险边界或不可生产状态的时间 |
| `subsystem` | 机械、液压、热、气氛、电气、流体 |
| `evidence` | 传感器、阶段、残差、漂移、变点、阈值距离 |
| `confidence` | 当前判断置信度，受数据质量和模型不确定性影响 |

## 4. 模型架构

### 4.1 数据治理层

职责：

- 从 InfluxDB 或离线文件读取数据。
- 对齐传感器采样。
- 标记停机段、缺失值和异常尖点。
- 根据 PLC 阶段信号切片。
- 生成训练、验证和回测样本。

### 4.2 基础表征层

候选模型：

- 开源时序基础模型：MOMENT、TSPulse、TTM、Chronos、TimesFM、UniTS。
- 轻量工程模型：TCN、Transformer Encoder、PatchTST、N-BEATS、AutoEncoder。
- 统计模型：稳健基线、EWMA、CUSUM、变点检测。

MVP 推荐：

- 用 TTM、Chronos 或 TimesFM 做预测残差基线。
- 用 MOMENT 或 TSPulse 做 embedding / reconstruction 基线。
- 同时保留稳健统计基线，作为解释和回退机制。

### 4.3 阶段感知层

工艺阶段必须进入模型或样本组织：

- 方式一：按阶段切片，阶段内单独建模。
- 方式二：将阶段作为 token 或外生变量输入模型。
- 方式三：先阶段切片，再在子系统模型中加入阶段 embedding。

MVP 推荐方式一和方式三结合。

### 4.4 预损坏风险融合层

风险分数建议由多类证据融合：

```text
R(device, subsystem, horizon)
= f(
    forecast_residual,
    prediction_uncertainty,
    reconstruction_error,
    baseline_drift,
    trend_slope,
    changepoint_score,
    physical_threshold_distance,
    cross_sensor_consistency,
    process_stage_context,
    data_quality
  )
```

初期 `f` 可以是可解释的规则模型或轻量打分模型；积累标签后再升级为监督学习模型。

## 5. 训练与校准策略

### 5.1 无故障标签阶段

训练目标：

- 预测未来正常传感器序列。
- 重构当前正常窗口。
- 学习设备和阶段状态 embedding。

校准方式：

- 使用早期稳定运行数据建立正常基线。
- 使用物理阈值事件作为弱标签。
- 使用专家复核结果标记有效风险事件。
- 使用在线影子运行统计误报。

### 5.2 有维修/故障记录阶段

新增目标：

- 预测未来 H 时间窗口内是否发生维修、停机或故障事件。
- 估计 RUL。
- 学习维护动作对风险回落的影响。

标签来源：

- 故障记录。
- 维修工单。
- 停机记录。
- 物理安全阈值越界。
- 专家确认的劣化事件。

### 5.3 多设备扩展阶段

策略：

- 共享通用 backbone。
- 每台设备保留设备 embedding 或 adapter。
- 每类设备建立传感器配置模板。
- 对新设备先冷启动基线，再逐步微调。

## 6. 解释机制

每个高等级风险输出都必须能回答：

- 哪个设备、哪个子系统、哪个传感器触发。
- 出现在什么工艺阶段。
- 偏离正常多少。
- 是否持续恶化。
- 是否接近物理危险阈值。
- 是否有多个相关传感器共同支持。
- 数据质量是否可靠。

解释不应只来自 LLM，而应来自模型和规则层的结构化证据。

## 7. 与应用系统边界

模型底座负责：

- 数据样本构建。
- 时序预测、重构、表征。
- 残差、漂移、变点和风险分数。
- 结构化证据输出。

预测性维护应用负责：

- 用户界面。
- 告警策略。
- 工单流转。
- 管理后台写入。
- 自然语言日报。
- 人工复核闭环。

LLM/Ollama 负责：

- 将结构化证据转成中文诊断摘要。
- 辅助生成日报和维护说明。

LLM 不负责核心风险判定。
