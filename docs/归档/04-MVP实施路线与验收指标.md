# MVP 实施路线与验收指标

版本：v0.1  
日期：2026-05-30

## 1. MVP 目标

MVP 的目标不是证明模型已经能准确预测所有设备故障，而是证明以下核心假设：

> 在真实设备时序数据上，阶段感知的时序预测和风险推演模型能够提前输出有解释证据的预损坏风险，并被预测性维护项目研发链路调用。

## 2. MVP 范围

### 2.1 设备范围

优先选择真空速凝炉 1-3 台设备。

选择标准：

- 数据连续性较好。
- PLC 阶段信号可用。
- 传感器覆盖机械、液压、热、气氛、电气或流体中的多个物理域。
- 项目团队能提供复核反馈。

### 2.2 子系统范围

建议优先级：

1. 液压系统：压力类传感器，容易结合物理阈值和执行阶段。
2. 气氛/真空系统：氧浓度，和工艺质量及密封风险强相关。
3. 热/冷却系统：温度、水流量、水温，适合趋势和效率衰退分析。
4. 机械系统：泵振动，适合健康劣化监测。
5. 电气系统：坩埚泄漏电流，适合高风险预警。

MVP 可先选择 2-3 个子系统，不必一次覆盖全部。

### 2.3 时间范围

- 实时或准实时窗口：最近数小时。
- 趋势窗口：最近 1 个月。
- 风险预测窗口：next_stage、next_batch、24h。

7 天以上窗口可作为发布阶段目标。

## 3. MVP 闭环

```text
数据接入
→ 数据质量检查
→ 工艺阶段切片
→ 训练/回测样本生成
→ 时序预测或重构
→ 残差与趋势计算
→ 预损坏风险融合
→ JSON 输出
→ 项目团队复核
→ 风险规则和模型校准
```

## 4. 工作包

### WP1：数据资产建设

产出：

- 设备、传感器、物理域、阶段配置表。
- 原始数据读取脚本或接口说明。
- 阶段化样本生成规范。
- 数据质量报告。

验收：

- 能稳定生成至少 1 台设备、1 个月数据的阶段化样本。
- 每个样本包含设备、传感器、阶段、时间窗口和数据质量标签。

### WP2：预测与表征模型评测

产出：

- 稳健统计基线。
- 至少 2 个开源模型的离线评测结果。
- 预测残差、置信区间、重构误差或 embedding 输出。

验收：

- 能在历史数据上完成回测。
- 能输出每个传感器/阶段窗口的模型误差和异常度。
- 能对模型效果、速度、部署复杂度做对比。

### WP3：预损坏风险融合

产出：

- 风险评分规则或轻量模型。
- 风险等级映射。
- ETA 计算方法。
- 证据链结构。

验收：

- 每条 WATCH 以上风险都有证据。
- 高等级风险能说明传感器、阶段、偏离幅度、趋势和阈值距离。
- 输出结果能被研发人员理解和复核。

### WP4：工程接口

产出：

- JSON Schema。
- 批处理或准实时推理入口。
- 图表数据输出。
- 模型版本和配置版本记录。

验收：

- 上层应用能读取模型结果。
- 同一输入数据重复运行结果可追溯。
- 失败时能给出数据质量或推理错误原因。

### WP5：复核与校准

产出：

- 风险事件复核表。
- 有效/无效风险标记。
- 阈值和融合权重调整记录。
- MVP 复盘报告。

验收：

- 项目团队完成至少一轮风险事件复核。
- 能识别主要误报来源。
- 能提出发布阶段需要补充的数据和标签。

## 5. 验收指标

### 5.1 必达指标

| 指标 | MVP 目标 |
| --- | --- |
| 数据可用性 | 至少 1 台设备 1 个月数据可被自动转成阶段化样本 |
| 模型输出 | 至少输出预测残差、趋势漂移、风险等级和证据链 |
| 风险窗口 | 支持 next_stage、next_batch 或 24h 中至少一个窗口 |
| 可解释性 | WARNING 以上结果必须包含传感器、阶段和偏离证据 |
| 工程接口 | 输出结构化 JSON，可被上层系统或脚本读取 |
| 复核机制 | 支持研发或项目团队标记风险事件是否有效 |

### 5.2 建议指标

| 指标 | 建议目标 |
| --- | --- |
| 影子运行 | 连续运行不少于 2 周 |
| 高等级误报 | 同一子系统高等级无效风险数量可控，并可解释原因 |
| 提前量 | 对专家确认的劣化事件，能在事件恶化或维护前给出提示 |
| 稳定性 | 推理失败、缺失数据、阶段异常时有明确降级策略 |
| 可迁移性 | 至少能在第二台同类设备上跑通数据和推理链路 |

## 6. 验证方法

由于初期没有故障标签，验证不能只依赖准确率。建议组合使用：

1. 回测预测误差：检验模型是否能学习正常运行轨迹。
2. 专家复核：对高风险片段进行有效/无效标记。
3. 规则事件验证：压力、氧浓度、泄漏电流等物理阈值附近事件。
4. 合成退化注入：人为注入漂移、突变、噪声和缓慢劣化，检验风险响应。
5. 在线影子运行：不触发真实工单，只记录模型风险和后续设备状态。
6. 维修记录对齐：后续如有维修和停机记录，用于校准风险提前量。

## 7. 时间建议

### 第 1 阶段：数据与基线

周期：1-2 周。

重点：

- 数据读取。
- 阶段切片。
- 稳健统计基线。
- 首版 JSON Schema。

### 第 2 阶段：模型评测

周期：2-3 周。

重点：

- 评测 TTM、MOMENT/TSPulse、Chronos/TimesFM 中的 2-3 个候选。
- 形成预测残差和重构异常输出。
- 确定 MVP 主模型和备用基线。

### 第 3 阶段：风险融合

周期：2 周。

重点：

- 风险等级映射。
- ETA 推演。
- 子系统风险聚合。
- 证据链输出。

### 第 4 阶段：影子运行与复核

周期：2-4 周。

重点：

- 连续运行。
- 风险事件复核。
- 误报分析。
- 发布阶段改进清单。

## 8. 发布阶段进入条件

满足以下条件后进入发布阶段：

- 数据、模型、风险、接口闭环已经跑通。
- 至少一个子系统的输出被项目团队认为有维护参考价值。
- 主要误报来源可解释，并有控制措施。
- 输出接口能够接入预测性维护项目。
- 有明确的模型版本、配置版本和运维边界。

## 9. 不建议纳入 MVP 的内容

- 一开始训练完全自研的大规模时序基础模型。
- 一开始承诺真实故障概率或精确 RUL。
- 一开始覆盖所有设备类型。
- 一开始让 LLM 参与核心风险判定。
- 一开始以管理层报告作为主要验收物。

这些内容可以进入发布或规模化阶段。
