B08 基础模型验证分析报告

本报告基于当前仓库的模拟数据、基础模型实验、TTM 本机推理报告和 code review 结果，回答三件事：模拟数据是否可以沉淀为真实数据治理规范，TTM 验证结果如何影响后续模型研发路线，以及项目应该沉淀哪些成果。

更新日期：2026-06-01 ｜ 证据来源：data/simulated/furnace_fu13_45d.parquet、reports/foundation_ttm_forecasting.md、src/b08_model_core/、tests/

结论摘要

TTM 已经在 FU13 模拟数据上完成真实本机 zero-shot forecasting，状态为 available_and_ran。在本轮 90 步上下文、16 步预测的短 horizon smoke test 中，TTM 的 MAE 为 1.190681、RMSE 为 2.368574，显著优于两个 baseline。当前应把 TTM 视为“短 horizon 直接引用候选”，同时尽快转入真实数据 schema map、真实数据 validation 和更多模型对比。

1. Code Review 与验证状态

review 审查结论

本轮审查未发现阻塞性代码问题。TTM 依赖预检、HF cache 环境隔离、离线 cache 命中、报告状态和 Git artifact 边界均已覆盖。

tests 自动验证

全量测试为 64 passed。TTM 使用 --no-download 复跑成功，证明本机 cache 可以支持离线推理。

artifact 本机权重边界

hf_cache/、模拟 parquet 和临时报告均被 Git 忽略。模型权重保持本机可用，不进入 GitHub。

2. Mock 数据合理性与数据标准

当前模拟数据不是随机构造的玩具表，而是围绕 FU13 真空速凝炉建立的“模型任务沙盘”。它保留了设备、批次、阶段、传感器、物理域、质量标记、弱退化标签和故障代理标签，因此可以直接倒推真实场景的数据治理规范。

维度当前模拟数据治理含义
设备与批次1 台设备 FU13，160 个 batch真实数据必须能稳定识别 device_id 和 batch_id，否则无法切分窗口和比较炉次。
采样与时间5 秒采样，时间范围为 2026-04-01 至 2026-05-04真实数据要保留原始时间戳、时区、采样间隔和停机间隔，不能只保留排序后的点位值。
阶段8 个工艺阶段阶段字典需要标准化；同一传感器在不同阶段的含义不同，不能把全流程混成一条无阶段曲线。
传感器16 个传感器，覆盖 6 个物理域需要传感器注册表：传感器 ID、单位、物理域、合理上下限、是否关键点位。
标签normal、incipient_degradation、risk_event、failure_proxy真实场景中维修、报警、阈值越界和停机记录可以先作为弱标签，不必等待完整故障闭环。

规范项建议要求验收方式

风险提醒

当前 mock 数据的质量标记全部为 OK，这适合第一轮模型链路验证，但不足以覆盖真实工业数据的缺失、维护、坏点和异步采样问题。真实数据接入时，质量标记和缺失处理应作为第一优先级。

3. 模型能力与效果判断

TTM 真实推理状态

available_and_ran。模型权重已下载到本机 hf_cache/，并可用 --no-download 离线复跑。

window 已验证窗口

context_length=90、prediction_length=16、16 个传感器。按 5 秒采样计算，约为 7.5 分钟历史预测未来 80 秒。

效果解释

相对 RobustStageForecaster，TTM 的 MAE 下降约 91.2%，RMSE 下降约 94.0%。这说明开源时序基础模型至少可以作为短 horizon forecasting 的强候选，而不是只能停留在依赖检查或概念验证。

后续研发路线推断

路线当前证据下一步动作Go/No-Go 条件
直接引用 TTM模拟数据短 horizon 显著优于 baseline，本机 cache 可离线复跑。迁移到真实 FU13 validation 后的数据；保留 90/16 配置做第一条真实数据 smoke test。真实数据上仍优于 baseline，且依赖和运行成本可接受。
轻量适配或微调当前 TTM 第一轮只消费数值通道和频率 token，阶段、传感器和物理域上下文尚未真正进入模型。测试阶段条件化、传感器分组、domain token adapter、少量真实数据微调。真实数据上 zero-shot 退化，或长 horizon/跨阶段任务需要领域上下文。
备用开源模型对比FlowState、TimesFM、Chronos、Moirai 已作为 fallback candidates 出现在报告中。复用同一 runner/report/status contract，逐个补 adapter。TTM 在真实数据或长 horizon 上不稳定，或部署约束不合适。
领域自研目前没有必要直接启动自研；开源模型已经证明 forecasting 链路有价值。先积累真实数据、任务定义和 benchmark，再判断是否需要小型设备时序基础模型。多个开源模型无法覆盖阶段语义、异常/补全/表征任务，且真实数据规模足够。

5. 建议的下一阶段

P0 真实数据治理闭环

拿一份真实 FU13 导出数据，完成 schema map、validation 和字段修正，确认真实数据能否进入同一套窗口构建。

P1 TTM 真实数据复测

先沿用 90/16 配置，在真实数据上跑 baseline 与 TTM，检查短 horizon 是否仍有优势。

P2 扩大任务和模型

扩展到长 horizon、跨阶段窗口、补全、表征和异常任务，同时接入 TimesFM、Chronos、Moirai 等模型。

本报告是阶段性研发判断，不代表生产部署结论。真实业务结论仍需真实 FU13 数据、更多设备样本和维修/报警记录验证。
