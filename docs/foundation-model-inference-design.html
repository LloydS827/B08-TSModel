B08 真实基础模型推理验证方案

本页把新阶段方案整理成中文可读版：先用可控的 FU13 模拟数据，在本机真正跑通一个开源时序基础模型，再用同一套指标判断直接引用、轻量微调、备用模型或自研路线。

更新日期：2026-06-01 ｜ 对应正式 spec：foundation model inference design

一句话结论

TTM 可以作为第一条最小可行链路，但项目架构不绑定 TTM、IBM、Hugging Face 或任何单一模型生态。真正要沉淀的是“模型无关的推理评测框架”：同一批窗口、同一组 baseline、同一套报告和同一套路线判断。

1. 为什么进入这个阶段

已经完成 模型核心沙盒已经可用

项目已经具备模拟数据、标准 schema、模型窗口、baseline、benchmark、真实数据 validation 和 forecasting 实验脚手架。

当前缺口 还没有真正运行外部模型权重

现有实验入口能检查候选模型依赖状态，但还没有完成真实 foundation model 的 zero-shot 推理。

新阶段目标 让模型路线判断有证据

只有模型真实跑起来，并和 baseline 同口径比较，才能判断是直接引用、轻量微调，还是继续寻找备用模型。

2. 本阶段做什么，不做什么

做什么

  • 使用已有 FU13 模拟数据构建模型窗口。
  • 本机安装可选依赖并下载开源模型权重。
  • 优先跑通一个轻量 forecasting 模型，例如 TTM。
  • 与 RobustStageForecaster 和 StageSeasonalNaiveForecaster 对比。
  • 输出包含指标、失败原因、依赖状态和路线建议的实验报告。
  • 在 README 中说明安装、下载、缓存和运行方法。

不做什么

  • 不扩展到告警平台、工单系统、日报系统或生产部署。
  • 第一版不要求一次性跑完所有候选模型。
  • 第一版不训练自研设备时序基础模型。
  • 不把模型权重、缓存、生成 parquet 或临时报表提交到 GitHub。
  • 不把 TTM 的输入格式写进 canonical schema。

3. 模型选择：首选一条链路，保留多条备用

优先级 候选模型 第一用途 价值 注意点
1 TTM / TinyTimeMixer zero-shot forecasting 轻量、forecasting 路线清晰、适合第一条本机推理链路。 偏预测任务，阶段/物理域语义覆盖有限。
2 FlowState / Granite Time Series forecast-first 备用 可作为 TTM 阻塞时的同类备用方案。 生态可能仍与 TTM 重叠。
3 TimesFM 强 forecasting 对照 适合检查趋势预测能力。 依赖和模型 API 不同，需要独立 adapter。
4 Chronos 概率预测对照 适合检查不确定性预测。 可能需要单变量或特定 reshape 策略。
5 Moirai / Uni2TS 多变量概率预测 适合后续作为多变量 forecasting 对照。 集成成本可能更高。
6 MOMENT / TSPulse / UniTS 表征、补全、异常后续评测 更贴近多任务基础模型能力。 不是最短的第一条 forecasting 链路。
关键约束

备用模型不是第一版自动全部实现，而是在报告中形成“下一步该试哪个 adapter”的建议。这样既保留弹性，也避免第一阶段被多个生态的依赖问题拖住。

4. 推荐架构

保留现有标准数据链路，把模型差异隔离在 adapter 内部。这样后续换模型时，不需要重写数据 schema 和窗口构建。

1模拟数据FU13 观测表 parquet
2窗口构建ModelWindow 列表
3Baseline稳健预测与季节 naive
4Adapter转换为模型输入
5模型推理本机权重 zero-shot
6报告决策指标、失败原因、路线建议
模块职责边界
FoundationForecastRunner 组织 baseline 对比、模型执行、状态处理和指标汇总。 不写模型私有逻辑。
FoundationForecastAdapter 定义依赖检查、模型加载、输入转换、预测和元数据接口。 不同模型通过不同 adapter 接入。
TTMForecastAdapter 第一版具体 adapter，负责 TTM 的导入、权重加载和推理。 不得污染 ModelWindow 和 canonical schema。
ForecastingExperimentReport 输出 Markdown 报告，服务研发和非技术评审。 既记录成功指标，也记录失败原因。

5. 命令形态

现有 forecasting 命令继续保留，新增模型选择、上下文长度、预测长度和缓存目录等参数。

uv run b08-model-core experiment forecasting \
  --dataset data/simulated/furnace_fu13_45d.parquet \
  --output reports/forecasting_ttm_experiment.md \
  --model ttm \
  --context-length 512 \
  --prediction-length 96 \
  --max-windows 40
场景退出码说明
baseline-only 成功0基础报告写出成功。
foundation model 成功0模型完成推理并写出报告。
缺依赖、缺权重、窗口不支持或运行失败1报告仍应写出，并记录失败原因。
非法命令参数2由 argparse 拒绝，例如长度小于等于 0。

6. 权重与缓存规则

允许 本机下载和使用模型权重

为了真正跑通模型推理，本阶段允许安装可选依赖，并把权重下载到 Hugging Face 默认缓存或指定本地目录。

禁止 模型权重不得进入 GitHub

需要忽略 models/、hf_cache/、.cache/、生成数据和临时报表。

README 必须写清楚安装和运行方法

包括 optional extra、缓存位置、下载行为、运行命令和报告解读方式。

7. 报告应该回答什么

报告内容为什么重要
数据集、窗口数、传感器数、上下文长度、预测长度确认不同模型是在同一任务上比较。
baseline 指标给开源模型提供最低对照线。
模型名称、adapter、checkpoint、缓存位置、依赖状态确保结果可复现，也能定位环境问题。
MAE、RMSE、覆盖率或分位数指标判断模型预测能力是否有实际价值。
失败状态和原因依赖失败、权重失败、窗口不支持也是路线判断证据。
IO 覆盖说明说明模型是否只吃数值序列，是否能表达阶段、传感器和物理域。
路线建议给出 direct reuse、few-shot、备用模型或 no-go 判断。

8. 如何做路线判断

实验结果下一步判断
真实模型明显优于 baseline，依赖成本可接受进入直接引用候选。
真实模型接近 baseline，运行稳定进入 few-shot 或冻结 backbone 轻量适配候选。
真实模型弱于 baseline，但成本低、链路稳定保留为对照模型，继续测试备用模型。
模型无法处理窗口形状或多传感器结构不适合作为直接引用路线，转测备用 adapter。
模型无法使用阶段、传感器、物理域信息只能算 forecast-only 候选，不能代表完整 B08 基础模型能力。
多个 forecasting 模型都失败或弱于 baseline优先补做表征、补全、异常模型评估，再考虑领域自研。

9. 第一版验收口径

最小可交付结果

至少完成一条真实 foundation model 推理链路，或输出清楚、可复现、可定位的失败报告。无论成功或失败，报告都必须能支持下一步模型路线判断。

  • baseline-only 模式仍可不依赖外部权重运行。
  • --model ttm 缺依赖时能写出清晰报告。
  • adapter 转换保持窗口数、上下文长度、预测长度和传感器顺序。
  • 报告包含状态、checkpoint、baseline 指标和路线建议。
  • README 写明权重本机保存、不提交 GitHub。
  • .gitignore 覆盖模型缓存和生成产物。

10. 关键风险

依赖变化 开源包和模型 API 可能变化

实现时需要再次核对官方安装说明，把模型私有逻辑封装在 adapter 内。

产物过大 模型权重可能很大

必须使用本地缓存，并通过 ignore 规则避免误提交。

预测成功不等于模型核心完成 forecast-only 不能覆盖全部任务

即使预测效果好，也要继续检查阶段、传感器、物理域、补全、表征和退化响应能力。

边界提醒

本页描述的是“底层时序模型能否真正跑起来并形成路线判断”。它不是完整预测性维护系统方案，也不承诺第一版直接产生生产告警、维修建议或工单。
