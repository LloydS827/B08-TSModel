# B08 Core Timeseries Model Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Define, prototype, and evaluate the B08 core equipment time-series foundation model by creating a realistic furnace simulation dataset, deriving model inputs/outputs from that dataset, and deciding whether to directly reuse, fine-tune, or train a model.

**Architecture:** Build a model-core research sandbox, not a complete predictive maintenance system. The work is organized into scenario simulation, task dataset construction, model IO specification, baseline and open-source model evaluation, route decision, and knowledge-output planning.

**Tech Stack:** Python 3.11+, numpy, pandas, scipy, scikit-learn, pydantic, pyarrow, pytest, matplotlib/plotly for diagnostics, optional PyTorch and Hugging Face for TTM/MOMENT/TSPulse/Chronos/TimesFM adapters.

---

## 0. Scope and Non-Goals

### In Scope

- Simulate one complete vacuum rapid solidification furnace scenario.
- Define the model-core input/output contract from the simulated and real-data constraints.
- Build model task datasets for forecasting, imputation, reconstruction, representation, degradation scoring, and future risk/RUL adaptation.
- Evaluate whether existing open-source time-series foundation models can be reused.
- Decide among direct reuse, light fine-tuning, and domain pretraining.
- Plan patent, paper, dataset, benchmark, and model-asset outputs.

### Out of Scope

- Front-end dashboard.
- Work-order system.
- Management backend integration.
- LLM-generated diagnosis reports.
- Production alarm rules.
- Complete predictive maintenance system delivery.

## 4. Route Decision Criteria

| Route | Enter When | Exit When | Minimum Evidence |
| --- | --- | --- | --- |
| Direct reuse | Candidate model supports required task and beats robust baseline without training | It cannot represent stage/domain context or deployment is too heavy | zero-shot/frozen metrics and adapter feasibility |
| Light fine-tuning | Candidate backbone is useful but domain gap is visible | Fine-tuning gain is small or unstable | linear probe/adapter/LoRA improves stage and degradation tasks |
| Domain pretraining | Open-source models fail core IO or degradation tasks | Data volume or compute is insufficient | custom pretraining objective beats baselines and has research novelty |
