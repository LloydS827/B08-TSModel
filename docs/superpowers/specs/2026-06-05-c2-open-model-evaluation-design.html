背景

C1 已经把 FU13 真实数据、baseline/TTM forecasting、representation、imputation 和结构化失败记录收束为第一版证据执行框架。C2 的职责是在此基础上进入开源时序基础模型的系统评测。

C2 的核心不是保证所有模型成功运行，而是让六个核心模型全部进入同一套审计、尝试和失败归因框架。

目标与非目标

目标

  1. 建立 C2 开源模型评测配置、运行入口、模型审计表、评测结果结构和 Markdown 报告。
  2. 将 TTM、MOMENT、Chronos / Chronos-Bolt、TimesFM、Moirai / Uni2TS、UniTS 全部纳入同一口径。
  3. 对 forecasting-first 模型执行 FU13 同窗口 forecasting 评测，或记录结构化失败。
  4. 对 representation / imputation / multi-task 模型执行 FU13 表征或补全评测，或记录结构化失败。
  5. 形成 C2 -> C3、C2 -> B 的阶段性决策记录。

非目标

  1. 不把 C2 做成公开数据集接入阶段；公开数据来源、许可证、schema mapping、标签语义和 split policy 进入 C3。
  2. 不进入 B 阶段自研模型训练。
  3. 不要求六个模型全部成功运行。
  4. 不将 TSPulse、FlowState 纳入 C2 核心必测集合。
  5. 不默认联网下载权重，不要求测试依赖真实 FU13 私有数据、网络或本机模型 cache。

架构

configs/c_stage_c1_execution.yaml
  + configs/c_stage_c2_open_model_evaluation.yaml
  -> C2 model registry
  -> C2 audit runner
  -> C2 task runner
     -> forecasting attempts
     -> representation attempts
     -> imputation attempts
  -> C2 evaluation results
  -> reports/c_stage_c2_open_model_evaluation.md

C2 复用 C1 的证据思路，但把中心从 evidence item 扩展为 model-task attempt。Registry 负责生成核心模型任务清单，audit runner 负责模型审计，task runner 负责成功运行或结构化失败记录。Audit record 应包含 source_ref 和 model_card_ref，使来源、仓库和权重说明可追溯。

状态语义

Audit Status

audit_passed来源、许可证边界、依赖、权重策略和任务接口已记录，未发现阻断
needs_license_review许可证或使用边界需要人工核对
needs_dependency_review依赖、安装方式或运行环境需要核对
needs_interface_review官方接口、adapter contract 或输入输出形状需要核对
audit_failed审计过程本身失败，原因必须记录

Model-Task Status

available_and_ran依赖、权重、输入形状和任务头可用，模型完成运行
missing_dependencyPython 包或运行时依赖缺失
missing_or_blocked_weights权重缺失、下载被禁用、下载失败或 cache 不可用
unsupported_taskadapter 或官方接口不支持当前任务
unsupported_window_shape当前窗口、变量数、horizon、mask 或 token 形式不被模型支持
runtime_failed运行时异常，已捕获并写入报告
license_or_interface_needs_review许可证或接口边界不足以安全执行，需要人工核对
skipped_by_configC2 配置主动跳过，默认核心模型不应出现该状态

报告与 CLI

建议新增本机报告 reports/c_stage_c2_open_model_evaluation.md，包含 report metadata、C2 scope、model audit table、model-task result matrix、forecasting section、representation / imputation section、baseline comparison、failure taxonomy、C3 handoff、B decision notes 和 invalid claims。

uv run b08-model-core experiment c-stage-c2 \
  --config configs/c_stage_c2_open_model_evaluation.yaml \
  --output reports/c_stage_c2_open_model_evaluation.md

默认不联网、不自动下载外部模型权重。单个候选模型失败不导致整个命令失败；配置、数据或报告错误应返回非零。

即使 audit 进入 needs-review 或 failed，系统仍应生成对应的 model-task attempt 并写入结构化失败。只有整体配置、FU13 评测窗口或报告输出不可用时，CLI 才应返回非零并终止。

测试策略

单元测试不得依赖真实 FU13 私有数据、网络或模型权重。测试使用小型 fixture、fake adapter 和 synthetic observations 验证 C2 行为。

  • C2 配置必须列出六个核心模型，缺一即失败。
  • registry 为每个核心模型生成至少一个主任务 attempt。
  • audit runner 能在依赖或权重缺失时生成 needs-review / failure record。
  • task runner 能记录成功、依赖缺失、权重缺失、unsupported task、unsupported window shape 和 runtime failed。
  • report renderer 必须包含六个核心模型、任务矩阵、失败分类、invalid claims、C3 handoff 和 B decision notes。
  • uv run python -m pytest -q 保持通过。

验收标准

  1. C2 配置固定列出 TTM、MOMENT、Chronos / Chronos-Bolt、TimesFM、Moirai / Uni2TS、UniTS 六个核心模型。
  2. C2 CLI 可以生成 Markdown 报告，且不要求六个模型全部成功运行。
  3. 每个核心模型都有 audit record。
  4. 每个核心模型至少有一个 model-task attempt，状态为成功或结构化失败。
  5. 报告包含 audit table、result matrix、failure taxonomy、invalid claims、C2 -> C3 handoff 和 C2 -> B decision notes。
  6. 默认项目状态保持可安装、可测试、可运行；C2 未完成模型成功不破坏 C1 已完成路径。
C2 的最终价值不是证明某个开源模型已经可生产使用，而是让六个核心候选在同一评测账本中接受审计、尝试和失败归因。
