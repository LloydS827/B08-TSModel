# 论文/专利主线深化设计

## 1. 背景

B08 已完成项目主线收束：第一目标是知识成果，包括论文、专利、学术综述和模型路线论证；第二目标是工程化产品成果，包括统一数据语料、开源模型适配、训练评测 workflow 和后续模型研发工作台。

当前项目不再缺少方向，而是需要把 A 阶段研究资产深化为可写作、可检索、可验证的知识成果包。下一阶段应以论文主线为牵引，专利方向与 C 阶段实验证据需求从论文主线中反推，避免过早进入大规模训练或再次堆积分散文档。

## 2. 目标

下一阶段目标是形成“论文框架优先型”的知识成果包设计，服务后续 A 阶段调研、C 阶段开源模型实验和 B 阶段条件性自训模型判断。

具体目标：

1. 明确近期论文主线：面向小样本工业物联设备健康管理的时序基础模型框架与验证。
2. 将论文主线拆成研究问题、相关工作、学术缺口、方法框架、验证路径和证据需求。
3. 从论文主线中拆出专利候选方向，并为每个方向定义技术问题、技术方案、关键步骤、输入输出、替代实现和证据需求。
4. 建立轻量但可追踪的调研资料组织方式，支持大量外部论文、模型卡、开源仓库和数据集资料的核对，而不让项目文档重新臃肿。
5. 反推 C 阶段最小实验需求，使后续模型适配和训练工作服务论文/专利证据，而不是孤立做工程实验。

## 3. 非目标

本阶段不做以下事情：

1. 不撰写完整投稿论文正文。
2. 不撰写正式专利申请文件或替代专利代理检索意见。
3. 不启动大规模模型训练。
4. 不新增大量按论文、模型或数据集拆分的碎片文档。
5. 不把 FU13 当前结果写成生产级预测性维护、生产告警、RUL 精确估计或自动维修建议能力。
6. 不把开源模型调研写成最终模型选型结论。

## 4. 推荐主线

近期论文主线建议固定为：

**面向小样本工业物联设备健康管理的时序基础模型框架与验证**

这条主线的定位是框架与验证型论文，而不是方法结果型论文。它优先证明以下内容：

1. 工业物联设备健康管理不同于普通 forecasting，需要阶段、传感器物理域、多任务、弱标签和小样本适配的统一研究框架。
2. 现有时序基础模型可以作为能力对照和适配起点，但在工业 metadata、弱标签任务、多任务表征和预测性维护目标任务上存在需要验证的缺口。
3. FU13 真机数据可作为第一真实闭环，开源预测性维护数据可作为第三层目标任务补充，两者需要通过统一 schema、任务层级和指标口径衔接。
4. 自研设备时序基础模型只能作为 A/C 阶段证据充分后的条件性方向，而不是预设结论。

## 5. 论文主线结构

下一阶段应把论文主线拆成以下模块：

| 模块 | 需要回答的问题 | 对应证据 |
| --- | --- | --- |
| 问题定义 | 为什么 B08 是小样本工业设备健康管理问题，而不是普通时序预测问题 | 学术主线综述、FU13 pipeline、任务指标矩阵 |
| 相关工作 | 主流时序基础模型、预测性维护、工业异常检测、弱标签时序学习分别解决了什么 | 论文、官方仓库、模型卡、数据集说明 |
| 学术缺口 | 现有路线在哪些方面不能直接覆盖 B08 | 开源模型矩阵、任务矩阵、数据矩阵 |
| 框架设计 | B08 的数据、任务、模型、评测和工程 workflow 如何组织 | 现有 research 资产、训练路线、产品化路线 |
| 验证路径 | 哪些任务现在可以验证，哪些必须等 C/B 阶段补证据 | FU13、开源数据候选、C 阶段实验设计 |
| 边界声明 | 哪些能力不能过度承诺 | 暂不主张能力、数据授权与专家复核需求 |

## 6. 专利方向结构

专利方向不单独发散，而是从论文框架中拆出技术方案候选。初始候选保留 5 个方向：

1. 工艺阶段与传感器物理域联合编码方法。
2. 工业设备小样本预测性维护的基础时序模型预训练方法。
3. 无故障标签条件下的设备异常候选信号生成方法。
4. 真机数据与开源设备健康数据融合的设备时序基础模型训练方法。
5. 基于多任务输出的设备状态表征、预测残差和退化趋势联合评估方法。

每个专利候选在执行阶段必须使用同一模板：

| 字段 | 含义 |
| --- | --- |
| patent_id | 候选编号，例如 `P1_stage_sensor_encoding` |
| technical_problem | 要解决的工业设备健康管理技术问题 |
| proposed_solution | 技术方案摘要 |
| key_steps | 关键步骤，避免只写概念 |
| inputs_outputs | 输入数据、模型输入、模型输出、评估输出 |
| alternatives | 可替代实现，帮助后续撰写权利要求层级 |
| evidence_needed | 需要哪些论文、实验、样例或专家复核支撑 |
| prior_art_keywords | 现有技术检索关键词 |
| current_status | candidate / evidence-needed / ready-for-disclosure |

## 7. 调研资料治理设计

本阶段涉及大量外部资料调研，但文档系统必须保持瘦身。执行时遵守以下规则：

1. 默认不为单篇论文、单个模型或单个数据集创建独立文档。
2. 外部资料只进入统一的证据登记表，不在多个文档重复摘抄。
3. 主线文档只保留与论文/专利论证直接相关的结论、缺口和引用 ID。
4. 原始 PDF、模型权重、数据下载包不得放入 `docs/`。
5. 只有当某类资料已经反复被多个主线模块引用时，才允许新增一个聚合文档。
6. 新增文档必须先说明为什么不能并入现有 `docs/research/` 文档。

推荐资料落库方式：

- `docs/research/paper-patent-directions.md` 作为论文/专利主线主文档，承接论文框架、专利候选和证据需求。
- 如外部来源数量超过 12 个，允许新增一个轻量文件 `docs/research/source-registry.md`，只保存来源登记表，不写长篇读书笔记。
- `docs/research/open-source-model-paper-matrix.md` 只在模型证据发生变化时更新，不重复承接论文主线全文。
- `docs/research/task-metric-matrix.md` 只在任务口径或指标门槛发生变化时更新。
- `docs/research/predictive-maintenance-dataset-matrix.md` 只在数据来源、授权或任务映射发生变化时更新。

`source-registry.md` 如需创建，应使用以下字段：

| 字段 | 含义 |
| --- | --- |
| source_id | 稳定编号，例如 `S001_TTM_2024` |
| type | paper / repo / model-card / dataset / patent / review |
| title | 标题 |
| url_or_doi | URL、DOI 或官方入口 |
| primary_source | 是否为论文、官方仓库、官方模型卡或数据集官方页 |
| relevance | 与 B08 的关系 |
| used_for | paper-background / model-gap / patent-prior-art / dataset-evidence / metric-design |
| key_claim | 仅记录与 B08 相关的最小证据，不做长篇摘录 |
| checked_at | 核对日期 |
| status | candidate / accepted / rejected / needs-review |

## 8. 外部调研原则

执行阶段必须优先核对 primary sources：论文原文、官方仓库、官方模型卡、数据集官方说明和专利公开文本。二手博客、新闻稿和模型排行榜只能作为线索，不能作为主要证据。

调研范围包括：

1. time-series foundation models。
2. industrial time series foundation model。
3. predictive maintenance foundation model。
4. self-supervised / weak-label industrial time series。
5. multivariate sensor representation。
6. anomaly candidate signal、residual-based monitoring、process monitoring。
7. RUL、run-to-failure benchmark 和 maintenance lead time。
8. 与阶段感知编码、多传感器物理域建模、数据融合训练、多任务设备状态评估相关的专利或现有技术。

每条引用必须能回答“它支撑哪个论文段落、哪个专利点或哪个实验需求”。不能回答这个问题的资料不进入项目文档。

## 9. C 阶段实验证据接口

论文/专利主线深化后，应反推出 C 阶段最小实验需求。实验需求不是本 spec 的执行对象，但必须被定义清楚，便于后续写 C 阶段计划。

建议证据接口：

| evidence_id | 支撑主线 | 所需实验 | 预期输出 | 禁止解释 |
| --- | --- | --- | --- | --- |
| E1_forecasting_residual | forecasting 是强基线但不足够 | FU13 TTM / forecasting-first 对照 | 预测误差、残差候选信号 | 不等于生产告警 |
| E2_representation | 工业设备需要状态表征 | MOMENT / UniTS embedding 或自研轻量表征对照 | embedding probe、stage/quality 弱标签结果 | 不等于真实健康等级 |
| E3_imputation | 多传感器缺失与重建是基础任务 | mask reconstruction / imputation | 重建误差、变量级结果 | 不等于异常识别 |
| E4_open_data_pm | 第三层任务需要 run-to-failure 或故障数据 | C-MAPSS / IMS / PRONOSTIA / TEP 候选验证 | RUL / fault / degradation 指标 | 不等于 FU13 已具备 RUL |
| E5_patent_effect | 专利技术效果需要样例支撑 | 阶段感知、多传感器、多任务对照 | 对照表、案例图、失败原因 | 不等于授权结论 |

## 10. 文档产物边界

本阶段完成后，文档系统应仍然保持清晰。默认允许修改或新增的文档范围如下：

| 文件 | 动作 | 说明 |
| --- | --- | --- |
| `docs/research/paper-patent-directions.md` | 更新 | 主承接文档，扩展为论文框架、专利候选、证据需求和调研路径。 |
| `docs/research/source-registry.md` | 条件性新增 | 仅当外部来源超过 12 个且放入主文档会显著臃肿时创建。 |
| `docs/research/open-source-model-paper-matrix.md` | 条件性更新 | 仅更新与模型证据相关的缺口、引用或候选分层。 |
| `docs/research/task-metric-matrix.md` | 条件性更新 | 仅更新与实验需求或指标门槛相关的接口字段。 |
| `docs/research/index.md` | 条件性更新 | 仅在新增 `source-registry.md` 或主文档状态变化时更新入口。 |
| `details.md` | 条件性更新 | 仅在阶段完成后记录一段摘要，不写流水账。 |

默认禁止新增以下类型文档：

1. `docs/research/papers/<paper>.md`。
2. `docs/research/models/<model>.md`。
3. `docs/research/patents/<patent>.md`。
4. 临时 HTML 报告。
5. 未纳入主线索引的散落 Markdown。

## 11. 成功标准

本阶段完成后，应满足：

1. 论文主线可以被概括为 1 个标题、1 个核心问题、3-5 个贡献点和一组证据需求。
2. 至少 5 个专利候选方向被结构化为技术问题、技术方案、关键步骤、输入输出和证据需求。
3. 外部调研资料有统一登记方式，且每条资料都能追踪到论文段落、专利点或实验需求。
4. C 阶段最小实验需求被反推出来，但不在本阶段启动实验实现。
5. 文档系统没有重新膨胀，新增文档数量不超过 1 个，除非用户明确批准。
6. 所有成果表述都保持审慎，不声称已完成生产告警、RUL 精确估计、自动维修建议、专利授权或自研基础模型训练成功。

## 12. Review 要点

Spec review 应重点检查：

1. 是否真正服务“知识成果优先，工程化产品第二”的项目主线。
2. 是否避免把论文/专利深化变成无边界资料堆积。
3. 是否明确了外部调研资料的落库路径和文档瘦身约束。
4. 是否保留 A -> C -> B 的阶段关系，而不是提前启动 B 阶段训练。
5. 是否给出了足够清楚的专利候选模板和 C 阶段证据接口。
6. 是否有过度承诺生产能力、专利授权或模型训练结果的风险。

## 13. 后续衔接

用户确认本 spec 后，下一步进入 writing-plans 阶段，产出执行计划。执行计划应优先安排：

1. 外部来源检索与 source registry 设计。
2. 论文主线结构化深化。
3. 专利候选模板填充。
4. C 阶段证据需求反推。
5. 文档索引最小更新与 review。

执行阶段应继续使用 subagent：一个或多个调研子代理负责外部 primary source 核对，主线程负责整合和文档瘦身边界，review 子代理负责检查主线一致性和过度承诺风险。
