# 项目主线收束与基础时序模型研发路线重构设计

## 背景

B08 项目已经完成几个关键阶段：

- 建立了设备时序模型研发沙盒，包括模拟数据、schema、窗口构建、baseline 和评测报告。
- 跑通了 FU13 真实多 CSV 数据到 canonical observations 的装配、验证和诊断。
- 在真实 FU13 窗口上完成 baseline 与 TTM 的同口径 forecasting 验证。
- 完成了 `leak_current_monitoring` 的第一版 scenario-filtered evaluation，把模型输出转化为 residual candidate signal。

这些工作证明项目已经具备基础时序模型研发所需的最小数据与评测闭环。但当前文档入口仍混合了早期方案、阶段过程、业务场景验证和模型研发路线，容易让后续研发者误以为项目主线是某个具体业务告警场景。

本轮重构只做文档叙事和研发路线收束，不做代码迁移。

## 项目重新定位

项目应定位为：

> 面向设备时序数据的基础模型研发与评测项目。

FU13 真实数据 pipeline、TTM 真实推理和漏液电流场景评测，是为了验证基础模型研发所需的数据标准化、窗口构建、模型输入输出、评测指标和候选信号映射能力。它们是基础设施和验证样例，不是项目最终目标。

项目主线是：

```text
设备时序数据
  -> canonical observations
  -> 模型窗口
  -> baseline / foundation model 评测
  -> residual / trend / spike / representation 等候选信号
  -> 支撑开源模型适配、轻量微调或自研基础模型训练决策
```

业务场景评测的角色应调整为：

- 作为基础模型输出进入业务语境的验证样例。
- 保留 scenario-filtered evaluation、质量过滤、等待态比较和 candidate signal 报告口径。
- 不再作为下一阶段唯一主线。
- 维修记录补充、生产告警闭环、现场维护策略等应用侧工作，可由工程团队继续承接。

## 本轮重构目标

本轮只处理 A 和 C 两类问题：

1. 文档叙事收束。
2. 研发路线收束。

本轮不处理代码模块迁移。

成功标准：

- README 的第一读者明确为研发执行者。
- README 能让研发者快速理解项目主线、当前可复现资产、标准运行入口和下一阶段研发路线。
- `details.md` 从流水进展转为阶段判断台账，说明已经证明什么、还没有证明什么、为什么路线转向基础模型研发。
- `docs/index.html` 从文档列表转为资料导航，区分主线入口、已验证资产、研究路线资料和归档资料。
- 下一阶段研发路线明确为 A -> C -> B：先调研，再系统适配开源模型，最后设计自研训练方案。
- 漏液电流场景被定位为验证样例，而不是项目终点。

## README 职责

README 面向研发执行者，而不是管理汇报读者。

README 应承担四个职责：

1. 说明项目主线：设备时序基础模型研发与评测。
2. 说明当前可复现资产：FU13 数据 pipeline、baseline/TTM 评测、scenario evaluation。
3. 给出标准运行入口：环境准备、真实数据装配、诊断、forecasting、scenario evaluation、测试。
4. 指向下一阶段研发路线：学术/行业调研、开源模型系统适配、自研训练方案设计。

README 应减少或下沉：

- 过长的历史叙事。
- 面向管理展示的阶段解释。
- 早期模拟阶段的详细背景。
- 已完成阶段的过程性设计。
- 不在当前主线中的成果规划。

README 不应删除历史资料，而应把它们导航到 `details.md`、`docs/index.html` 或归档目录。

## details.md 职责

`details.md` 保留为项目进展台账，但应从“流水账”调整为“阶段判断台账”。

它应重点说明：

- 当前项目已经证明了什么。
- 当前还没有证明什么。
- 为什么项目路线从业务场景闭环转回基础时序模型研发。
- 哪些工作由工程团队继续承接，例如维修记录、生产告警和现场业务闭环。
- 为什么下一阶段需要补学术资料、行业资料和开源模型系统对比。

`details.md` 可以继续面向非技术人员解释项目状态，但不再反向主导 README 的结构。

## docs/index.html 职责

`docs/index.html` 应从“按生成时间堆放文档”调整为“按用途导航资料”。

建议分为四组：

```text
主线入口
  - README
  - details
  - 项目主线与研发路线收束设计

已验证资产
  - TTM 真实数据能力复核报告
  - 漏液电流场景评测报告
  - 真实数据 schema / validation 说明

研究与路线资料
  - 开源时序基础模型调研
  - 行业/学术资料综述

```

这样后续研发者可以先看主线入口，再进入可复现实验和研究资料，而不是被历史文档淹没。

## 下一阶段研发路线

下一阶段路线按 A -> C -> B 推进。

### A. 学术 / 行业 / 模型路线调研

目标：回答设备时序基础模型应训练什么、评测什么，以及它与通用时间序列模型的差异。

建议产物：

- 设备时序基础模型任务谱系：forecasting、imputation、representation、anomaly、classification、RUL 等。
- 工业设备数据特点：多传感器、工艺阶段、非均匀采样、维护事件缺失、工况切换、质量标记。
- 学术与行业资料综述：基础时序模型、工业时间序列、设备健康管理、预测性维护相关路线。
- 开源模型能力矩阵：TTM、MOMENT、Chronos、TimesFM、Moirai、UniTS 等模型按任务能力、依赖成本和输入输出限制归类。
- 训练目标候选：masked reconstruction、next-window forecasting、contrastive representation、stage-aware prediction、multi-task learning。
- 数据规模判断：现有 FU13 数据能支持什么，不能支持什么；哪些数据需求属于模型研发，哪些属于工程应用。

完成标准：

- 能说明为什么需要或不需要自研设备时序基础模型。
- 能给出开源模型优先评测列表。
- 能给出本项目专属的任务/指标/数据需求清单。

### C. 开源基础时序模型系统适配与对比

目标：判断已有基础模型能否作为本项目核心基座。

建议产物：

- 统一 adapter contract。
- 统一窗口与指标口径。
- 多模型可运行性验证。
- 按任务维度比较 forecasting、embedding、imputation、anomaly response。
- 依赖成本、部署难度、推理速度和缓存策略记录。
- Go / No-Go 结论：直接引用、轻量微调、还是进入自研。

完成标准：

- 不只比较模型名字，而是在同一批设备窗口、同一套指标、同一套报告口径下比较。
- 能解释模型失败是依赖问题、窗口形状问题、任务不匹配，还是模型能力不足。
- 能给出是否进入自研训练方案设计的证据。

### B. 自研设备时序基础模型训练方案设计

目标：在 A 和 C 之后，设计最小自研训练路线。

本阶段不应一开始就大规模训练，而应先形成可审查的训练方案。

建议产物：

- 模型输入格式。
- 预训练目标。
- 数据集组织方式。
- 训练/验证切分。
- 与 baseline 和开源模型的对照口径。
- 最小训练原型计划。
- 算力、数据规模和维护成本估算。

完成标准：

- 能说明自研模型相对开源模型要解决的具体缺口。
- 能说明最小训练原型的成功标准。
- 能说明继续投入训练的 Go / No-Go 条件。

## 归档与保留边界

保留在主线入口：

- FU13 real data pipeline。
- canonical observation schema。
- window builder。
- baseline / TTM evaluation。
- scenario evaluation。
- 当前测试与复现命令。
- 学术/行业/开源模型路线资料。

降级为支撑或归档：

- 早期模拟数据展示型文档。
- 旧 HTML 实施方案。
- 已完成阶段的过程性计划。
- 面向管理展示的早期路线材料。
- 尚未进入当前阶段的专利/论文成果规划。

本轮不删除历史材料，只调整入口层级和叙事权重。

## 不做代码重构的边界

本轮不做：

- 不移动 `src/b08_model_core` 目录结构。
- 不拆 `real_data/scenario_evaluation.py`。
- 不改 CLI 命令。
- 不删旧 adapter。
- 不改测试结构。
- 不做模型训练实现。

原因：当前主要问题是项目主线叙事和研发路线尚未收束，而不是代码结构已经阻碍开发。先完成 A/C 收束，再判断是否需要代码层重构。

## 实施建议

后续实施计划应按文档影响面由高到低推进：

1. 重写 README 的定位、当前状态、标准流程和下一阶段路线。
2. 更新 `details.md`，把当前阶段判断改为基础时序模型研发路线收束。
3. 重组 `docs/index.html` 的导航分组。
4. 新增或更新研究路线资料入口，承接 A 阶段调研。
5. 标记旧方案和过程性文档的归档位置。

实施过程中仍需遵守：

- 不提交真实数据、生成 parquet、real reports、模型 cache。
- 不扩大代码重构范围。
- 不把业务场景验证写成项目终点。
- 不把自研训练写成已经决定的路线；它是 A/C 之后的条件性路线。
