背景

B08 已经完成 FU13 真实数据 pipeline、baseline / TTM 同口径 forecasting、漏液电流 residual 样例、A 阶段研究资产和 C0 最小证据契约。C1 的职责是把这些前期工作收束为第一版可执行证据体系。

C1 不应只保守完成单点 E1，也不应扩展成通用实验平台。它应同时推进 E1 真实实验闭环、E1-E5 可复用的最小证据框架、E2/E3 模型扩展入口。

目标与非目标

目标

  1. 将 E1_forecasting_residual 推进到可运行、可报告、可审计的真实实验闭环。
  2. 建立 C 阶段最小 evidence framework，统一配置、registry、执行结果、模型状态和报告字段。
  3. 将 E2_representation 和 E3_imputation 纳入同一执行框架。
  4. 为下一阶段系统评测 TTM、MOMENT、Chronos、TimesFM、Moirai / Uni2TS、UniTS 预留接口。
  5. 保持文档系统克制，只新增本 Markdown 与同名 HTML 阅读版。

非目标

  1. 不进入 B 阶段自研模型训练。
  2. 不做泛化实验平台或生产级预测性维护系统。
  3. 不接入公开预测性维护数据集；E4 保持 C2 候选方向。
  4. 不把 residual、probe 或 reconstruction error 解释为生产告警、FU13 RUL、自动维修建议或专利授权结论。
  5. 不要求测试依赖真实 FU13 私有数据、外部网络或模型权重。

范围决策

层级范围验收强度

架构

configs/c_stage_minimum_evidence.yaml
  + configs/c_stage_c1_execution.yaml
  -> C1 evidence registry
  -> C1 evidence runner
     -> E1 forecasting residual
     -> E2 representation probe
     -> E3 imputation / reconstruction
  -> C1 evidence results
  -> reports/c_stage_c1_evidence_report.md

C1 执行配置只记录本轮执行参数，不复制 C0 的完整证据表。Registry 负责继承 C0 的证据边界，Runner 按 E1/E2/E3 调度现有能力，Report Renderer 输出统一证据账本。

统一结果结构必须包含 evidence id、experiment id、task id、status、dataset boundary、split policy、model results、primary metrics、failure reasons、artifact outputs、invalid claims 和 decision gate notes。

任务路径

E1 forecasting residual

FU13 canonical observations
  -> C1 window policy
  -> baseline forecasting
  -> optional TTM forecasting
  -> residual metrics and top-k examples
  -> E1 report section

E1 是 C1 的锚点。baseline 路径必须可运行；TTM 在依赖和 cache 具备时真实运行，否则输出结构化失败。

E2 representation probe

FU13 windows
  -> statistical embedding baseline
  -> optional MOMENT / UniTS embed adapter
  -> probe labels with input exclusion note
  -> probe or clustering metrics
  -> E2 report section

E2 重点是表征口径和输入排除。若 stage、quality_flag 或 failure_proxy 被作为输入 metadata，报告必须限制 probe 的解释边界。

E3 imputation / reconstruction

FU13 multivariate windows
  -> deterministic mask strategy
  -> simple reconstruction baseline
  -> optional MOMENT / UniTS imputation adapter
  -> reconstruction metrics by sensor and mask type
  -> E3 report section

mask 策略固定 seed、mask ratio、mask type 和评测窗口范围，只在评测窗口内生成。

状态语义

状态含义
available_and_ran模型完成运行
missing_dependencyPython 包或运行时依赖缺失
missing_or_blocked_weights权重缺失、下载禁用或 cache 不可用
unsupported_taskadapter 不支持当前任务
unsupported_window_shape窗口形状、变量数、horizon 或 mask 形式不支持

planned_not_executedC0 已规划但 C1 不执行，例如 E4/E5

CLI 与错误处理

uv run b08-model-core experiment c-stage-c1 \
  --config configs/c_stage_c1_execution.yaml \
  --output reports/c_stage_c1_evidence_report.md

默认不联网、不自动下载外部模型权重。E1 baseline 无法运行、C0 contract 无法读取、C1 配置无效或报告无法写出时返回非零。候选模型失败时仍写报告，并由配置决定是否严格返回非零。

测试策略

单元测试使用小型 fixture 或 synthetic observations，不依赖真实 FU13、网络或模型权重。测试覆盖配置继承、registry 校验、result schema、失败状态报告、probe 输入排除、deterministic mask、CLI exit code 和默认 pytest。

验收标准

  1. 存在 C1 CLI 命令，能从 C1 配置生成 Markdown 报告。
  2. E1 baseline 路径真实运行，并输出 forecasting metrics、residual summary、top-k examples 或明确失败原因。
  3. TTM 状态被记录为 available_and_ran 或结构化失败状态。
  4. E2 有 statistical embedding baseline、probe 输入排除说明、候选模型状态和报告分节。
  5. E3 有 deterministic mask、simple reconstruction baseline、候选模型状态和报告分节。
  6. 报告继承 C0 invalid claims，并明确 C1 不构成生产告警、RUL、自动维修或专利授权结论。
  7. E4/E5 在 C1 报告中显示为未执行承接项，不被误写成完成。
  8. 默认测试通过，且测试不依赖私有数据、外部网络或模型权重。
