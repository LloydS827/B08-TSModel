# 业务场景评测口径固化设计

## 背景

B08 当前已经完成从 FU13 原始多 CSV 到 canonical observations、模型窗口、baseline/TTM forecasting 报告的工程闭环。这个结果证明项目已经具备基础模型评测和适配的研发底座，但还没有证明模型输出可以直接变成维修建议、故障概率、RUL 或生产告警。

下一阶段的核心不是马上横向接入更多模型，也不是直接训练自研模型，而是把基础模型输出和业务场景之间的评测桥梁建立起来。只有先明确模型应该看哪些数据、在哪些业务场景里评测、输出如何转成候选异常信号，后续模型选择、轻量微调或业务闭环才有可信依据。

## 目标

本阶段目标是固化第一个业务场景的评测口径，让项目能够回答三个问题：

1. 在选定业务场景中，模型应该看哪些传感器、哪些工艺阶段、哪些时间窗口。
2. 基础模型的 forecasting 误差和残差，哪些可以转化为候选异常信号。
3. 在缺少真实故障标签和维修记录时，哪些结论只能称为候选异常，哪些可以进入专家复核或后续维修记录对齐。

成功后，项目应从“TTM 在真实窗口上能跑通并优于简单 baseline”推进到“可以围绕一个业务场景判断基础模型输出有没有业务分析价值”。

## 非目标

本阶段不做以下事情：

- 不声称设备故障预测、RUL、维修建议或生产告警已经完成。
- 不把 forecasting 误差直接解释为设备故障。
- 不同时深做所有业务场景。
- 不优先接入 TimesFM、Chronos、Moirai、FlowState 等更多模型。
- 不训练或微调基础模型，除非评测口径已经证明 frozen/zero-shot 路线存在明确缺口。
- 不改动原始 `data/real/` 文件，也不提交生成 parquet、临时 reports 或模型 cache。

## 概念解释

### 窗口质量治理

基础模型不是直接读取整个月数据，而是读取一段段时序窗口。窗口质量治理就是判断这些窗口是否有业务意义。

如果一个窗口混入等待态、无效值、未分配 cycle、异常采样间隔或跨阶段混合片段，模型误差可能升高。但这不一定说明模型差，也不一定说明设备异常。治理窗口质量的意义是先确认评测对象可信，避免把数据切片问题误判为模型能力问题或设备状态问题。

### Scenario-filtered evaluation

当前 scenario 指标主要是“跑完模型后按传感器映射分组看误差”。这只能说明某类传感器上的误差情况，不能说明模型真的在某个业务场景下被评测。

真正的 scenario-filtered evaluation 应该先按业务场景筛选窗口，再评测模型。例如液压系统检测应重点关注浇筑阶段的压力响应，气氛检测应关注溶解和浇筑阶段的氧含量变化，漏液电流监测应关注生产阶段中的电流趋势和尖峰。它的意义是把问题从“模型整体预测好不好”推进到“模型对某个业务问题有没有用”。

### 质量标记过滤

当前真实数据包含 `good`、`invalid`、`unassigned_cycle` 等质量标记。质量标记过滤要比较不同数据质量口径下的模型表现，例如 `good only`、去除 `invalid`、去除 `unassigned_cycle`。

它的意义是区分数据质量导致的误差、正常工艺波动导致的误差，以及可能具有业务意义的异常候选信号。

### 等待态处理

`上盖开启` 等等待态可能不是生产核心过程。如果等待态进入 forecasting 窗口，模型可能学到等待、空转或切换过程，而不是设备运行规律。

等待态处理需要评估保留、剔除或单独建模三种口径。它的意义是明确哪些时间段参与模型评测，哪些时间段只作为上下文或单独状态处理。

### 更强 baseline

baseline 是判断基础模型有没有价值的最低门槛。当前 robust 和 seasonal naive baseline 已经提供基础对照，但下一阶段需要更强的工程对照，例如 rolling、lag、分位数、按传感器/阶段的简单统计模型或轻量机器学习 baseline。

它的意义是避免只因为 TTM 是基础模型就高估其价值。只有当基础模型在清晰业务场景和强 baseline 下仍然有优势，后续模型扩展或微调才值得投入。

本阶段的最小强 baseline 不引入复杂机器学习模型，先限定为一个按传感器的 rolling 或 lag baseline，并与现有 `RobustStageForecaster`、`StageSeasonalNaiveForecaster` 同口径比较。轻量机器学习 baseline 只作为后续阶段候选，不进入本阶段成功标准。

## 推荐推进方式

本阶段采用单场景最小闭环，而不是同时深做四个场景。

第一场景选定为 `leak_current_monitoring`。选择理由：

- 该场景只有 `LeakElec` 一个核心传感器，第一轮窗口和残差解释更容易收束。
- 该场景适合探索预测残差、尖峰和趋势是否能形成候选异常信号。
- 当前 TTM 评测中该场景指标有明显改善空间和解释价值。

`atmosphere_detection` 作为第二候选场景保留，只有当漏液电流监测闭环稳定后再复制评测口径。

本阶段完成以下闭环：

```text
业务场景定义
-> 相关传感器和工艺阶段
-> 有效窗口规则
  -> baseline/TTM 同口径评测
  -> 残差或误差分布分析
  -> 候选异常信号定义
-> 专家复核或维修记录对齐需求
```

## 数据流

现有数据流保持不变：

```text
data/real/
  -> real-data assemble-fu13
  -> canonical observations
  -> real-data diagnose-fu13
  -> model windows
  -> baseline/TTM forecasting
```

新增的是业务场景评测层：

```text
canonical observations
  -> scenario window selection
  -> quality/waiting filters
  -> baseline/TTM comparison
  -> forecasting residual and candidate-signal report
```

这个层次应尽量复用现有 `real_data.forecasting`、`window_builder`、baseline 和 TTM adapter，不为了单场景实验重写整条 pipeline。

## 阶段交付物

本阶段最小交付物应包括：

- 一个选定业务场景的场景评测设计。
- 可复用的窗口筛选规则，包括传感器、阶段、质量标记和等待态处理。
- 同一场景下 baseline 与 TTM 的评测报告。
- 候选异常信号报告，说明哪些信号来自 forecasting 残差、趋势、尖峰或分位数阈值。
- 明确的边界说明：当前只是候选异常，不是故障结论。

如果第一个场景验证有效，再复制到其他场景；如果第一个场景发现评测口径不稳，应先修正口径，而不是接入更多模型。

## 成功标准

本阶段完成时应满足：

- 能解释选定业务场景为什么选择这些传感器、阶段和窗口。
- 能比较 `all data`、`good only`、去除 `invalid`、去除 `unassigned_cycle` 等质量口径的指标变化。
- 能说明等待态保留、剔除或单独处理对模型误差的影响。
- 能在现有 baseline、一个 rolling 或 lag baseline 和 TTM 之间进行同口径比较。
- 能把模型输出转成候选异常信号，并说明该信号还需要哪些业务证据才能进入维护决策。
- 能给出下一轮模型扩展、轻量微调或业务数据补充的明确优先级。

## 推荐路线

推荐路线是：

1. 先用 `leak_current_monitoring` 做单场景最小闭环。
2. 固化该场景的窗口筛选、质量过滤、等待态处理和 baseline 对照。
3. 将 TTM 输出转换为 forecasting 残差、趋势或分位数候选信号。
4. 如果信号可解释，再扩展到其他场景和更多模型。
5. 如果信号不可解释，优先补充数据标注、维修记录或强 baseline，而不是直接微调模型。
