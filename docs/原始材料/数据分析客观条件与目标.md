## 一、业务场景

**真空速凝炉**是一种用于制备快淬合金磁粉的工业设备，其运行过程涉及高温熔炼、真空环境控制、机械传动、冷却系统等多个物理子系统。设备一旦发生故障，不仅造成单炉次产品报废，还可能引发连锁停机损失。

本系统的核心诉求是：**在故障实际发生之前，通过传感器数据的持续分析，识别设备健康状态的劣化迹象，给出预警和可执行的维护建议。**

## 二、客观约束条件

### 2.1 数据层面的约束

| 约束 | 说明 |
|-----|------|
| **零故障标签** | 系统上线初期没有任何历史故障记录可供学习。无法采用"先标注故障、再训练分类器"的经典监督学习范式。 |
| **数据量小** | 单台设备的历史数据通常只有 1~2 个月，累计炉次几十到上百次。不支持需要大量样本训练的复杂模型。 |
| **采样非均匀** | 传感器采样周期约 5 秒，但炉次之间可能存在长时间停机，数据密度在"运行期"和"停机期"差异巨大。 |
| **无周期性** | 生产不固定班次、不固定炉次间隔，不存在可依赖的日/周/季节性周期模式。 |
| **设备个体差异** | 同型号不同炉号的设备，其传感器正常读数范围可能存在显著差异，无法用统一的绝对阈值覆盖。 |

### 2.2 物理层面的约束

| 约束 | 说明 |
|-----|------|
| **多物理域耦合** | 振动（机械）、温度（热）、压力（液压/真空）、氧浓度（气氛）、电流（电气）等多个物理量同时变化，可能存在相互关联。 |
| **工艺阶段强相关** | 同一传感器在不同工艺阶段（如"抽真空"vs"浇筑"）的物理意义完全不同，不能混为一谈。 |
| **隐含状态** | 某些关键过程（如"产品放入冷却容器"）没有直接的传感器信号，需要通过温度曲线形状间接推断。 |
| **物理安全阈值有限** | 仅少数传感器有明确的物理安全上下限（如压力不超过 12 MPa），大多数参数没有行业统一异常标准。 |

### 2.3 工程层面的约束

| 约束 | 说明 |
|-----|------|
| **内网部署** | 系统运行在工厂内网，无法依赖云端 API 做核心计算（LLM 等辅助功能除外）。 |
| **有 GPU** | 有一台 GPU（5090*2） 服务器，可用于深度学习模型的训练与推理。 |
| **实时性要求适中** | 不需要毫秒级响应，分钟级分析周期即可接受。 |
| **可解释性要求高** | 现场运维人员和管理层需要理解"为什么报警"，黑箱模型难以落地。 |

## 三、可用输入

### 3.1 传感器时序数据

从设备采集器持续写入时序数据库的原始数据：

- **时间戳** + **数值**（字段名为 FValue）
- **设备标识**（measurement 名，如 FU13_Pump_01）
- **传感器标签**（Attribute 字段，如 PumpShake1、SysSelfPressure）
- **采样周期**：约 5 秒/点

**当前接入的传感器类型**（按物理域划分）：

| 物理域 | 传感器 | 物理含义 |
|-------|-------|---------|
| 机械 | 泵振动（2 支） | 机械泵轴承/泵体健康 |
| 液压 | 系统压力、炉盖压力、比例阀压力、坩埚前倾/回程压力 | 液压系统密封性与执行机构状态 |
| 热 | 冷却容器外壁温度（3 支）、线圈回水温度（4 支）    | 冷却效率与热负荷分布 |
| 气氛 | 熔炼室氧浓度、冷却筒下料口氧浓度 | 真空密封性与气氛控制 |
| 电气 | 坩埚泄漏电流 | 坩埚壁绝缘状态（击穿风险前兆） |
| 流体 | 冷却筒水流量/水温、急冷辊水流量/水温 | 冷却水路通畅性 |

### 3.2 工艺阶段信号

从 PLC 控制器输出的离散工艺阶段序列，映射为 8 个阶段：

`上盖开启 → 上盖关闭 → 抽真空 → 浇筑 → 溶解 → 测温 → 冷却 → 氩气导入`

该信号用于：
- 区分传感器在不同工艺阶段下的物理上下文
- 将传感器数据按阶段切分，分别建立正常基线
- 识别目标阶段的特定异常模式（如"抽真空末端氧含量"）

### 3.3 设备配置信息

静态配置数据，包括：
- 设备清单与传感器映射关系
- 传感器类型归属（振动/温度/压力等）
- 物理安全阈值（上下限，部分传感器有）
- 故障知识库关联 ID（用于异常发生时关联已知故障模式）
- 管理系统设备参数 ID（用于上报关联）

## 四、期望输出

系统需要产出四类输出，覆盖"实时告警—趋势推演—诊断建议—管理闭环"的完整链路。

### 4.1 实时异常检测结果

**目标**：对最近数小时内的传感器数据进行分析，判断当前是否存在异常。

**期望输出的数据结构**：

```
- 设备标识
- 传感器标识
- 分析时间窗口范围
- 异常等级（四级）：
    NORMAL（正常）
    NOTICE（注意）
    WARNING（警告）
    DANGER（危险）
- 可生产性评级（四级）：
    A：正常运行
    B：监控运行（统计偏移，可继续）
    C：计划维护（持续异常，安排停机窗口）
    D：立即处理（物理安全阈值越界）
- 异常特征快照（触发异常的统计量及其偏离程度）
- 逐窗口/逐段的异常时序（供前端绘制趋势图）
```

### 4.2 长周期趋势推演结果

**目标**：对最近 1 个月的数据进行分析，识别传感器特征的长期演化趋势，预判未来风险。

**期望输出的数据结构**：

```
- 趋势线标识（设备 + 传感器 + 特征维度）
- 早期基线统计量（中位数、稳健标准差）
- 当前值相对基线的漂移量（以 σ 为单位）
- 趋势显著性（斜率方向 + 统计检验 p 值）
- 变点记录（何时发生状态突变）
- ETA 外推（若趋势持续，预计多久到达危险阈值）
- 健康等级（四级：NORMAL / WATCH / WARNING / ALERT）
```

### 4.3 设备级综合诊断报告

**目标**：将单传感器的结果聚合成设备视角的综合报告，供管理层快速决策。

**期望内容**：
- 设备整体健康等级
- 各子系统（机械/液压/热/气氛/电气/流体）的风险总览
- 异常参数清单及严重性排序
- 可生产性判断与生产建议
- 维护优先级与排查建议
- 条件化的故障时间预估（"若当前趋势持续..."）

### 4.4 管理闭环数据

**目标**：将分析结果写入管理后台，形成可追溯的运维记录。

**期望输出的数据接口**：

```
- 分析记录（每条传感器每次分析一条记录，含完整报告 JSON + 图表地址）
- 预测故障（当异常达到 WARNING 及以上时，自动生成预测故障条目，
            关联故障知识库，预计故障时间）
- AI 诊断日报（按设备按天汇总，含自然语言诊断摘要）
```

## 五、核心问题的本质

将上述条件与目标抽象后，本系统的本质是一个**在强约束下的无监督工业时序异常检测与趋势推演问题**：

| 维度 | 本质 |
|-----|------|
| **无监督** | 没有故障标签，无法训练分类器，只能学习"什么是正常" |
| **上下文敏感** | "正常"的定义随工艺阶段、设备个体差异而变化 |
| **多时间尺度** | 既需要小时级的实时异常捕捉，也需要月级的慢变趋势跟踪 |
| **可解释性** | 输出必须能被现场人员理解并转化为行动 |
| **多物理域** | 需要整合振动、热、流体、气氛、电气等不同物理量的信息 |

## 六、与其他系统的边界

| 系统 | 职责边界 | 本系统的输入/输出关系 |
|-----|---------|---------------------|
| **数据采集层**（PLC + 采集器 → InfluxDB） | 负责传感器数据的采集与存储 | 本系统**输入**：从 InfluxDB 读取原始数据 |
| **管理后台**（.NET WebAPI） | 负责设备档案、故障知识库、工单管理 | 本系统**输出**：POST 分析记录与预测故障；**输入**：设备参数 ID、故障知识库 ID |
| **前端大屏**（Next.js） | 负责数据可视化展示 | 本系统**输出**：JSON 报告 + PNG 图表供其读取 |
| **AI 大模型服务**（Ollama） | 负责自然语言生成 | 本系统**调用**：将结构化报告转换为中文诊断摘要 |
