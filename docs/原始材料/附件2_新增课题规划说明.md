## 01 三步判断

### 第一步：符合战略

说明课题如何服务海陆空立体无人制造体系、大型复杂构件空间制造、AI 赋能决策与自动化，以及与 iClaw、rClaw、物理数据层、工业模型或应用智能体的关系。

### 第二步：课题成立

说明真实问题、依据来源、目标用户、技术不确定性、最小闭环和边界。无法形成最小闭环的内容，先作为模块、依赖或素材管理。

### 第三步：

说明技术成果和经营成果。技术成果包括研发对象 TRL、IP、论文、数据、评测和文档；经营成果包括产品、模块、方案和样板项目。

## 03 PEA 填写要求

| 评审环节 | 填写要求 |
| --- | --- |
| 问题定义 | 拿出具体依据，说明要解决什么问题。依据可以来自客户、项目、现场、产品或技术瓶颈。 |
| 目标定义 | 用户是谁；目标只能有一个且必须是结果；要求可有 1-3 个，作为实施方案约束。 |
| 方案描述 | 说明上线后的用户操作体验和最小闭环；实现方案结论先行，弱化技术细节，突出方案差异。 |
| 风险与措施 | 写关键技术风险及应对措施，必要时写继续、收窄、调整、合并、冻结或终止条件。 |
| 资源评估 | 只写影响课题成败的关键资源，包括人员、数据、设备、客户协同、外部授权和上线条件。 |
| 成果衡量 | 写核心指标及预期值，并说明与目标达成的关系；同时写何时验证、未达预期如何应对。 |
| 竞品分析 | 可选。可写竞品、参考案例、借鉴点和差异化，但不能用竞品代替问题定义。 |

## 04 四阶段证据

| 阶段 | 重点问题 | 最低证据 |
| --- | --- | --- |
| 想法 | 课题为什么值得做，是否符合战略，问题和成果是否清楚。 | 课题定义、PEA 初稿、目标用户、最小闭环和成果预设。 |
| MVP | 核心假设是否成立，是否能在真实或半真实场景运行。 | 可运行 Demo、关键数据、测试记录、用户反馈和风险清单。 |
| 发布 | 0-1 研发是否结束，是否可以作为产品或交付能力进入经营。 | 发布版本、验收指标、交付包、运维边界和成果台帐记录。 |
| 规模化 | 是否具备复制、销售、运维和持续改进能力。 | 多项目复用、客户案例、收入/价值证据和标准化资产。 |
