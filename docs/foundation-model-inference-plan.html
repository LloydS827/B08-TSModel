B08 真实基础模型推理实施计划

本页是可执行 Markdown plan 的中文可读版。目标是在 FU13 模拟窗口上真正跑通至少一个开源时序基础模型，优先从 TTM 开始，并用同一套指标和 baseline 判断下一步是直接引用、轻量微调、备用模型，还是转向领域自研。

更新日期：2026-06-01 ｜ 对应完整计划：Foundation Model Inference Implementation Plan ｜ 对应方案：真实基础模型推理验证方案

一句话执行目标

新增一个模型无关的 foundation forecasting 边界，让 baseline-only 流程继续不依赖外部权重，同时让 TTM 在本机可选安装、可选下载、可真实推理，并输出能和现有 baseline 同口径比较的报告。

1. 关键判断门槛

必须保留 基础流程不依赖模型权重

默认 `uv sync --extra dev` 和 `uv run pytest -q` 仍应能运行，不要求下载 TTM、Hugging Face 权重或大型缓存。

第一条真实链路 TTM 先跑通，但不锁死生态

TTM 是第一轮最小可行 adapter。TimesFM、Chronos、Moirai、FlowState 暂不实现，只在报告里作为后续备用路线。

本机权重 模型权重只保存在本机

Hugging Face cache、模型权重、生成 parquet 和临时报告不能提交到 GitHub。README 需要写明下载、缓存和使用方法。

完成标准 available_and_ran 才算真实跑通

missing_dependency、missing_or_blocked_weights、runtime_failed 只能说明链路未完成，不能作为阶段通过。

2. TTM 下载设计审查结论

安装依赖、指定缓存、允许下载是三件不同的事

uv sync --extra dev --extra foundation-ttm 只安装 Python 包，不代表模型权重已经下载。--model-cache-dir hf_cache 或 HF_HOME=hf_cache 只说明权重读写位置。只有 --allow-download 才允许本次运行从 Hugging Face 下载缺失权重。

运行情况 报告状态 命令结果
baseline 模式 skipped_by_user 或 baseline-only 报告 退出码 0
选择 TTM，但 optional 依赖未安装 missing_dependency 退出码 1，仍写出报告
选择 TTM，依赖已安装，--no-download，权重已在本机 cache 中 available_and_ran 退出码 0
选择 TTM，依赖已安装，--no-download，本机 cache 没有权重 missing_or_blocked_weights 退出码 1，仍写出报告
选择 TTM，依赖已安装，--allow-download，下载、加载和推理成功 available_and_ran 退出码 0
选择 TTM，但下载、加载或推理失败 missing_or_blocked_weights 或 runtime_failed 退出码 1，仍写出报告
实现要求

HF_HOME 和 HF_HUB_OFFLINE 只应在 TTM runtime 调用期间临时设置，并在调用后恢复。--model-cache-dir 是本次运行的显式覆盖项，不应使用 setdefault 让旧环境变量悄悄抢优先级。

3. 实施路径

1 结果契约 统一状态、指标、失败原因和路线建议。
2 报告输出 把 baseline comparison、cache 状态和模型状态写进 Markdown。
3 TTM adapter 延迟导入依赖，转换 FU13 窗口到 TTM 可预测格式。
4 Runner 和 CLI 统一生成窗口、运行 baseline、运行模型、生成报告。
5 本机验证 安装 optional extras，下载权重，确认 TTM 真实完成推理。

5. 任务拆解

Task 1: Foundation Result and Metric Contract

目标：先把结果对象和指标口径定下来，让模型成功、失败、缺依赖、下载受阻、运行失败都有明确状态。

  • 新增 FoundationModelStatus 和 FoundationForecastResult。
  • 给 forecasting metrics 增加 RMSE，并支持只有点预测、没有区间预测的模型。
  • 实现 recommend_route，根据模型状态和 baseline 对比给出 direct reuse、fine-tune candidate、no-go 等建议。

验证重点：

  • available_and_ran 的 succeeded 为 true。
  • missing_dependency 不会被误判为成功。
  • RMSE、MAE、interval coverage 同时可测。
Task 2: Markdown Report Renderer

目标：生成一份能服务路线决策的 Markdown 报告，而不是只输出模型分数。

  • 报告包含 Dataset Summary、Selected Foundation Model、Baseline Comparison、Local Model Environment、Route Recommendation。
  • baseline delta 必须写清楚，例如 TTM 相对 RobustStageForecaster 的 MAE/RMSE 差异。
  • 记录 optional dependency、cache 目录、权重状态和失败原因。

验证重点：

  • 成功报告能看见 available_and_ran。
  • 失败报告能看见明确原因，但不被当成完成。
  • 报告适合研发和非技术评审共同阅读。
Task 3: TTM Adapter Dependency and Input Conversion

目标：让 TTM 的依赖检查和输入转换先可测，不要求测试阶段真的下载外部权重。

  • 检测 granite-tsfm、torch、transformers、huggingface_hub 是否存在。
  • 把 ModelWindow 转成 TTM 能消费的 pandas 格式。
  • 保留 sensor/stage/domain 等上下文的可追踪信息，即使第一轮 TTM 只使用数值列。

验证重点：

  • 缺依赖时返回 missing_dependency。
  • 窗口形状、预测长度和通道数能被明确检查。
  • adapter 内部封装 TTM 特有逻辑，不污染 canonical schema。
Task 4: TTM Lazy Real-Inference Boundary

目标：把真实模型加载、权重下载和推理边界放进 adapter，并用 allow_download 明确控制网络行为。

  • 默认不下载权重，避免基础测试被外部网络和 Hugging Face 状态影响。
  • 当 --allow-download 开启时，允许本机下载并缓存模型权重。
  • 对下载受阻、模型加载失败、推理失败分别给出状态和原因。

验证重点：

  • 所有重量级 import 都是 lazy import。
  • 下载被禁止且 cache 未命中时返回 missing_or_blocked_weights，并在 reason 和 weight_status 中说明。
  • adapter 不吞掉错误，报告能看到可诊断的失败原因。
Task 5: Foundation Runner and Baseline Comparison

目标：新增 FoundationForecastRunner，统一运行现有 baseline 和选定 foundation adapter。

  • 复用现有窗口构建和 baseline，不重写数据入口。
  • 把 foundation model 和 RobustStageForecaster、StageSeasonalNaiveForecaster 做同口径比较。
  • 输出 baseline deltas，支持后续路线判断。

验证重点：

  • fake adapter 成功时 runner 能生成完整比较结果。
  • fake adapter 失败时 baseline 指标仍可输出。
  • baseline-only 默认路径不被破坏。
Task 6: Experiment Entry Point and CLI Options

目标：把真实模型实验接进现有 b08-model-core experiment forecasting，并保持老命令可用。

  • 新增 --model baseline 和 --model ttm。
  • 新增 --allow-download、--model-cache-dir、--context-length、--prediction-length。
  • 模型未成功跑通时，CLI 需要返回非零退出码，防止误把失败报告当完成。

验证重点：

  • 默认命令仍是 baseline-only。
  • TTM 缺依赖时报告写出，但命令失败。
  • 参数校验继续拒绝不合理窗口配置。
Task 7: Optional Dependencies, Cache Ignore Rules, and Lockfile

目标：把 TTM 依赖放到 optional extra 中，并确保本机 cache 不进入 Git。

  • 在 pyproject.toml 增加 foundation-ttm optional dependencies。
  • 更新 uv.lock，让依赖解析可复现。
  • 在 .gitignore 忽略 hf_cache/、model_cache/、reports/foundation_*.md 等本机产物。

验证重点：

  • uv sync --extra dev 不安装大模型依赖。
  • uv sync --extra dev --extra foundation-ttm 才安装 TTM 所需包。
  • 权重、cache 和生成数据不出现在提交中。
Task 8: README and Progress Ledger

目标：把下载、运行、缓存、本机权重和报告解释写进项目入口。

  • README 说明如何安装 optional TTM 依赖。
  • README 写清楚模型权重只保存在本机，不上传 GitHub。
  • details.md 更新能力、风险和近期进展，让非技术人员能理解阶段变化。

验证重点：

  • 文档包含本机 cache 说明。
  • 文档包含成功和失败状态解释。
  • 当前阶段判断没有被夸大成生产可用。
Task 9: Manual TTM Local Verification

目标：在本机真正安装 optional 依赖、下载权重，并运行 TTM 推理。这个任务是本阶段最关键的实证门槛。

uv sync --extra dev --extra foundation-ttm
HF_HOME=hf_cache uv run b08-model-core experiment forecasting \
  --dataset data/simulated/furnace_fu13_45d.parquet \
  --output reports/foundation_ttm_forecasting.md \
  --model ttm \
  --allow-download \
  --model-cache-dir hf_cache \
  --max-windows 20

验证重点：

  • 报告中必须出现 available_and_ran。
  • 报告中必须出现 Baseline Comparison 和 Local Model Environment。
  • 若失败，需要记录原因并把计划状态保留为未完成。
Task 10: Final Verification Gate

目标：在声明完成前运行完整验证，并检查 Git 状态，确保只提交源码、测试和文档。

uv run pytest -q
git diff --check
git status --short

验证重点：

  • 所有测试通过。
  • 没有 whitespace error。
  • 没有权重、cache、parquet 或临时报告进入提交。
  • 最终回复明确 TTM 是否真实跑通，以及报告路径。

7. 什么情况下不能算完成

失败报告有价值，但不是完成标准

如果最终状态是 missing_dependency、missing_or_blocked_weights、runtime_failed、unsupported_window_shape 或其他非 available_and_ran 状态，本阶段只能说明发现了阻塞点，不能宣布“真实基础模型推理已跑通”。

不能只看模型名称

TTM、TimesFM、Chronos、Moirai、FlowState 都需要回到同一批窗口和同一组指标比较。只有真实推理结果优于或接近 baseline，且适配成本可接受，才进入 direct reuse 或轻量微调候选。
