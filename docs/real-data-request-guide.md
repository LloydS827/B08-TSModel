# FU13 真实数据提供说明

本文档用于向现场或数据系统负责人说明：当前需要什么样的 FU13 真实数据，以及原始数据如何交付给 B08 设备时序基础模型项目组。

## 核心建议

请优先提供一份真实、完整、未人工清洗的 FU13 原始时序数据导出，并同时提供字段说明和导出背景说明。

原始数据可以直接给我们。现场不需要为了模型项目提前补齐、平滑、删除异常值、重命名字段或手工修正数据。我们希望尽量保留真实现场状态，包括缺失、重复、异常值、停机段、维护段、报警段和采样不稳定等问题。

这些问题不是噪声，而是后续建立数据治理规范、质量标记规则和模型输入标准的重要依据。

## 推荐对外说明

可以直接向现场或系统负责人发送以下说明：

```text
我们现在需要一份 FU13 设备的真实时序数据，用于验证设备时序基础模型的数据接入、schema 映射和预测效果。

数据不要求提前清洗，原始导出也可以。我们希望尽量保留现场真实状态，包括缺失、重复、异常值、停机段、维护段、报警段等信息，后续由我们统一做数据质量检查和治理规则整理。

建议提供以下内容：

1. 一份 FU13 原始时序数据导出文件
   - 格式可以是 CSV、Excel、Parquet 或数据库导出。
   - 长表或宽表都可以。
   - 时间范围建议至少覆盖若干个完整生产批次/炉次；如果方便，优先给 1 到 2 个月数据。
   - 不需要人工补齐、平滑、删除异常值或改字段名。

2. 字段说明
   - 每一列的含义。
   - 时间字段是哪一列，时间格式是什么，是否带时区。
   - 设备编号字段是哪一列，例如 FU13。
   - 批次/炉次字段是哪一列；如果没有批次字段，请说明如何从工艺记录中判断批次边界。
   - 工艺阶段字段是哪一列，例如抽真空、氩气导入、溶解、测温、浇筑、冷却等。
   - 每个传感器字段或测点 tag 的含义、单位和物理域，例如温度、压力、真空度、电流、流量、振动等。

3. 质量与事件信息，如果有的话请一并导出
   - 传感器缺失、无效、维护、停机等质量标记。
   - 报警记录。
   - 维修记录。
   - 停机或异常生产记录。
   - 质量判定、返工、报废、人工备注等弱标签信息。

4. 数据导出说明
   - 数据来源系统名称。
   - 采样频率，例如 1 秒、5 秒、10 秒或不固定采样。
   - 是否存在重采样、聚合、补点或系统自动清洗。
   - 是否存在权限脱敏、字段裁剪或单位换算。
```

## 最低交付内容

| 内容 | 是否必须 | 说明 |
| --- | --- | --- |
| FU13 时序数据文件 | 必须 | CSV、Excel、Parquet、数据库导出均可。 |
| 字段说明 | 必须 | 至少说明时间、设备、炉次、阶段、测点、读数的来源字段。 |
| 测点单位 | 强烈建议 | 后续需要转换为模型中的 `unit` 和 `domain`。 |
| 批次/炉次信息 | 强烈建议 | 没有批次字段时，需要说明如何从时间或工艺记录重构批次。 |
| 工艺阶段信息 | 强烈建议 | 没有阶段字段时，需要提供阶段事件表或工艺开始结束时间。 |
| 报警、维护、停机记录 | 可选但重要 | 可作为退化标签、故障代理标签和数据质量标记来源。 |
| 数据导出说明 | 必须 | 说明来源系统、采样频率、时间范围、是否已被系统处理。 |

## 可接受的数据形态

### 长表

长表是一行一个测点读数，通常适合历史数据库或时序数据库导出。

| ts | equipment | lot | phase | tag | reading |
| --- | --- | --- | --- | --- | --- |
| 2026-05-01 08:00:00 | FU13 | B20260501001 | 抽真空 | VacuumPressure | 0.42 |
| 2026-05-01 08:00:00 | FU13 | B20260501001 | 抽真空 | PumpShake1 | 1.20 |

### 宽表

宽表是一行一个时间点，每个测点是一列，通常适合 Excel 或报表系统导出。

| ts | equipment | lot | phase | VacuumPressure | PumpShake1 | OutletOxygen |
| --- | --- | --- | --- | --- | --- | --- |
| 2026-05-01 08:00:00 | FU13 | B20260501001 | 抽真空 | 0.42 | 1.20 | 0.08 |

两种形态都可以。我们会通过 `configs/real_data_schema_map.template.yaml` 将原始导出映射到项目标准 schema。

## 标准 schema

真实数据和模拟数据最终都会统一成以下观测表字段：

| 标准字段 | 含义 |
| --- | --- |
| `timestamp` | 采样时间 |
| `device_id` | 设备编号，例如 FU13 |
| `batch_id` | 生产批次、炉次或重构批次 |
| `stage` | 工艺阶段 |
| `sensor_id` | 规范化测点 ID |
| `value` | 数值读数 |
| `unit` | 工程单位 |
| `domain` | 物理域，例如 mechanical、thermal、atmosphere、electrical、fluid、hydraulic |
| `quality_flag` | 数据质量标记，例如 good、missing、invalid、maintenance |
| `degradation_label` | 退化弱标签，默认 normal |
| `failure_proxy` | 故障代理标签，布尔值 |

## 字段说明模板

现场可以按以下格式补充字段说明：

| 原始字段名 | 字段含义 | 示例 | 单位 | 备注 |
| --- | --- | --- | --- | --- |
| ts | 采样时间 | 2026-05-01 08:00:00 | - | 请说明时区 |
| equipment | 设备编号 | FU13 | - | 如果字段名不同，请填写真实字段名 |
| lot | 炉次/批次 | B20260501001 | - | 如果没有，请说明如何判断炉次边界 |
| phase | 工艺阶段 | 抽真空 | - | 请提供阶段枚举或阶段事件表 |
| VacuumPressure | 真空压力 | 0.42 | kPa | 示例测点 |

## 我们接收后的处理方式

拿到数据后，项目组会按以下顺序处理：

1. 保留原始文件，不直接覆盖或手工修改。
2. 建立真实数据 schema map，将原始字段映射到标准字段。
3. 运行真实数据 validation，检查字段缺失、时间解析失败、未知测点、未映射阶段、非数值读数和重复点。
4. 形成数据质量报告，区分数据问题、字段说明问题和模型输入问题。
5. 将可复用规则沉淀为真实场景数据治理规范。
6. 在 validation 通过后，使用同一批数据运行 baseline 和 TTM 等基础模型评测。

## 注意事项

- 不要为了“看起来干净”提前删掉异常点。异常点可能正是模型需要学习或识别的对象。
- 不要只导出少量已经筛选过的正常片段。模型评测需要覆盖正常、波动、停机、维护和异常生产状态。
- 如果涉及脱敏，可以保留设备编号、测点 ID 和批次 ID 的稳定替代值，但不要每次导出随机替换。
- 如果部分字段暂时没有，也可以先给原始数据。缺失字段会在 validation 报告中明确列出，再决定补充方式。
