# 预测性维护与设备健康数据集矩阵

## 目的

本矩阵用于筛选能支撑小样本工业物联设备健康管理时序基础模型的真实或公开数据，服务 B08 的知识成果沉淀和后续模型训练路线设计。

需要强调的是，本文件是候选数据集矩阵，不代表其中数据已经全部下载、清洗、映射或完成授权。后续如进入实际训练、论文、专利或产品化验证阶段，应对每个数据集的来源、许可证、下载方式、再分发限制和可用任务边界逐项核对。

## 数据筛选原则

第一阶段优先选择与预测性维护、设备健康、剩余寿命预测（RUL）、故障诊断和退化过程直接相关的数据。这类数据更接近 B08 关注的工业物联设备健康管理场景，也更适合验证小样本条件下的弱标签学习、退化趋势建模和跨设备泛化能力。

暂不优先纳入金融、交通、天气、电力负荷等通用时间序列数据。它们可以作为时间序列建模方法的补充参考，但不应替代设备健康管理主线数据，也不应作为预测性维护能力的第一证据。

筛选时建议优先关注以下条件：

- 是否来自真实设备、工业过程或被广泛引用的设备退化实验。
- 是否包含 RUL、run-to-failure、故障类型、退化阶段、维护事件或可构造弱标签的信息。
- 是否包含多变量传感器序列，便于映射到统一的设备时序 schema。
- 是否能支持小样本、跨工况、跨设备或跨域迁移评测。
- 是否具备清晰的数据许可、引用要求和下载来源。

## 与 FU13 的关系

FU13 是当前可控真机数据和 pipeline 验证入口，应作为 B08 真实工业设备数据路径的第一样例。它的核心价值不只是提供传感器时间序列，更包括现场语义、工艺阶段、数据质量、弱标签构造和真实设备 pipeline 的可落地验证。

公开数据集用于补足 FU13 暂时不一定覆盖的任务形态，例如 RUL、run-to-failure、明确故障分类、退化趋势和跨设备泛化验证。它们可以帮助建立更完整的训练与评测矩阵，也可以为论文和知识成果提供可复现实验入口。

公开数据不能替代 FU13 的现场语义。即使公开数据在标签完整性或 benchmark 可比性方面更强，也不能直接证明 B08 已具备生产级预测性维护能力；它们应被定位为补充验证、方法对照和预训练/微调路线探索的数据来源。

## 统一 schema 映射问题

为了将 FU13 与公开数据纳入同一训练和评测框架，后续需要建立统一 schema。第一版 schema 至少应考虑以下字段：

核心 observation-level 字段用于描述单条或单窗口观测记录，应尽量保留可直接进入训练样本构造的最小语义：

- `timestamp`
- `device_id`
- `batch_id / run_id`
- `stage` 或 `operating condition`
- `sensor_id`
- `value`
- `unit`
- `domain`
- `quality_flag`
- `degradation_label`
- `failure_proxy`
- `RUL / maintenance_event if available`

dataset-level metadata 用于记录数据来源、文件粒度、映射状态和标签置信度，不一定逐行重复写入训练样本，但应能追溯每条样本的来源和映射过程：

- `source_dataset/source_file`
- `cycle_id` 或 `batch/run/cycle` 关系
- `sampling_rate/time_index_type`
- `mapping_status/label_confidence`
- `license_status`
- `source_status`
- `split_policy`
- `version/download_date if available`

不同数据集的字段含义和粒度可能存在明显差异。例如，run-to-failure 数据通常以一次退化实验或一个 engine/unit 作为 `run_id`；工业过程故障数据可能更关注故障编号、注入时间和工况；FU13 则需要保留现场阶段、批次、质量标记和弱标签语义。

因此，统一 schema 不应简单追求字段名一致，而应明确每个字段的语义来源、可置信程度和是否为弱标签。对于无法可靠映射的字段，应保留为空、标注为 unknown，或记录在数据集级 metadata 中，避免在训练阶段引入伪精确标签。

## 训练与评测用途

FU13 用于验证真实工业设备 pipeline 和弱标签任务，包括数据接入、清洗、质量标记、阶段识别、failure_proxy 构造以及面向现场语义的时序表示学习。

开源 run-to-failure 数据用于补足 RUL、故障分类和退化趋势任务。它们适合作为模型能力边界的可复现实验入口，例如退化趋势预测、剩余寿命估计、异常检测、跨工况评测和小样本微调。

模拟数据或退化注入数据只适合作为弱标签和冷启动补充，不应作为第一证据。它们可以帮助调试 schema、pipeline、训练脚本和评测协议，但不能单独证明模型已经具备真实设备预测性维护能力。

建议将训练与评测用途分为三个层级：

- 真实 pipeline 验证：以 FU13 为主，确认数据链路、字段语义和弱标签任务是否可运行。
- 公开 benchmark 验证：以 C-MAPSS、IMS Bearing、PRONOSTIA / FEMTO-ST、Tennessee Eastman Process 等为候选，验证 RUL、退化、异常和故障分类能力。
- 冷启动与方法调试：使用模拟或退化注入数据辅助调试，但不作为生产能力或核心知识成果的唯一依据。

## 后续矩阵接口

为了让本数据集矩阵能够承接 `task-metric-matrix.md` 和 `foundation-model-training-route.md`，后续每个数据集条目应补充面向任务和训练路线的接口字段。这些字段不表示数据已经完成下载、授权、清洗或映射，而是用于判断某个数据集是否可以进入下一阶段。

- `task_family`：描述数据集可支持的任务族，例如弱标签设备健康评估、RUL、run-to-failure 退化预测、故障分类、异常检测或跨设备泛化。该字段应与 `task-metric-matrix.md` 中的任务分类保持一致。
- `allowed_metrics`：记录在当前标签和评测协议下允许使用的指标，例如分类指标、异常检测指标、趋势误差指标或 RUL 误差指标。若标签语义不清晰，应标记为待核对，避免使用不适配的精确 RUL 指标。
- `training_role`：标记数据在训练路线中的角色，例如真实 pipeline 验证、预训练候选、微调候选、评测候选、冷启动补充或仅用于方法调试。该字段应与 `foundation-model-training-route.md` 中的训练阶段衔接。
- `split_policy`：记录训练、验证、测试划分策略，例如按设备划分、按 run 划分、按时间划分或按工况划分。对于 run-to-failure 数据，应避免同一设备或同一退化轨迹泄漏到多个评测集合。
- `go/no-go prerequisite`：记录进入下一阶段前必须满足的前置条件，例如许可证核对通过、来源可追溯、字段映射完成、标签置信度可接受、评测协议明确、FU13 现场语义不被公开数据替代等。

这些接口字段的作用是把“候选数据能否使用”拆成可审计条件，而不是直接承诺某个公开数据集可以进入正式训练。只有当来源、授权、schema 映射、任务族和评测指标都清楚后，才适合把对应数据纳入训练路线或对外成果描述。

## 许可证和下载风险

候选数据是否可下载、是否允许再分发、是否可用于论文、专利或模型训练，需要逐项核对。不同数据集可能存在引用要求、注册下载、研究用途限制、再分发限制、商业使用限制或原始链接变更等问题。

本矩阵不等于数据授权清单，也不表示 B08 已经取得这些数据的再分发权或训练使用权。任何进入正式实验、论文、专利材料、开源仓库或产品演示的数据，都应先完成来源记录、许可证核对、引用格式确认和内部合规判断。

在下载和使用外部数据前，建议至少记录：

- 官方来源或可信镜像链接。
- 数据许可或使用条款。
- 是否允许再分发原始数据或派生数据。
- 是否允许用于论文、专利、模型训练和商业场景。
- 需要保留的引用、致谢或声明。
- 数据版本、下载日期和校验信息。

对于许可不清晰、下载入口失效或用途边界不明确的数据，应先标记为“待进一步核对”，不得默认纳入训练主线或对外成果描述。
